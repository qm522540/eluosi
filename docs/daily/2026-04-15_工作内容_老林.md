## 二、Ozon 数据源采集的 3 个深坑（今日最大收获）

### 坑 1：Performance API 两条硬限制

用户反馈"曝光/点击/花费全为 0"，查日志发现全是 400 Bad Request 或 429。花了两轮才从 Ozon 返回的**俄语**错误消息里挖出真相：

| 限制 | 错误码 | 俄语原文 |
|---|---|---|
| 单次 submit 最多 10 个活动 | 400 | `Превышен лимит по количеству кампаний (максимум 10)` |
| 同时只能 1 个活跃 UUID（前一个必须完成） | 429 | `Превышен лимит активных запросов (максимум 1)` |

我之前的"全量 18 个一次 submit" 或 "5 个/批 × 4 批并发" 都会踩。修为 `batch ≤ 10` + 严格串行（`await _submit_and_fetch` 内部等 UUID 到 OK 才 return）（`0d5e8d0` `277fa8b`）

### 坑 2：stat_date 取错字段

Performance API 响应的默认 `createdAt` 字段是 **SKU 创建时间**（常为 `"0001-01-01"` 或无关历史日期），不是广告展示日期。默认 `statistics/json` 还是整区间汇总，根本没日级数据。

修为请求加 `groupBy: "DATE"`，让 Ozon 按天 × SKU 拆分返回每行带 `date` 字段（`DD.MM.YYYY` 俄文格式）（`9209277`）

### 坑 3：活动列表同步漏了新活动

用户平台 Excel 04-14 有活动 `24664750` 共 4 个 SKU 广告数据，但我们 DB 完全没这活动。Ozon API 实际返回 20 个，我们 DB 只有 18 个。

- `14341569` 是系统级 SEARCH_PROMO，跳过合理
- `24664750` 是 04-13 刚创建的新活动，Celery 定时任务没来得及把它入库

今天手工补了 `24664750` 进 DB（`fetch_ad_campaigns` + `_upsert_campaign` 直接调用）。

### 坑 4：Seller API 订单伪归因

`_fetch_seller_data` 拿到店铺所有订单（含自然+广告），代码 hardcode 塞给 `campaigns[0]`。导致 DB 里 `15679268` 出现 9 条 `imp=0 orders>0` 的"假广告行"，和 Ozon 平台后台完全对不上。

修为 smart_sync **彻底不再调 Seller API**，`ad_stats` 所有字段（orders/revenue 也）由 Performance API 提供（Ozon `groupBy=DATE` 响应本身有 `orders + ordersMoney` 字段，是广告真实归因）（`d92ccec`）

### 最终验证
用户 Ozon 后台导出的 04-14 共 17 条广告数据，**逐行逐列 100% 一致**。再拉 35 天结果：295 条 / 35 天，曝光 210,702、点击 10,845、花费 ₽29,870.04、订单 273、营收 ₽144,756。
