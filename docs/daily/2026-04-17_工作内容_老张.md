## 七、下午追加：地区销售决策闭环（4 轮迭代）

用户问"地区销售能看出投产比吗？投产不高的地区不投广告"——引出完整调研 + 实现。

### 7.1 方案 A：净贡献估算 + 屏蔽建议（`2547bdb`+`3b1d42f`）

**核心限制**：WB/Ozon 广告花费**没有地区拆分**（平台 API 根本不给），严格的地区 ROAS 做不出来。

做了可以代替的决策视图：
- 后端 `ranking` 加字段：`net_profit_est`（销售额 × 店铺毛利率 − 退货损失）+ `suggestion`（block/watch/keep）+ `suggestion_reason`
- 店铺毛利率三级回退：products.net_margin 平均 → ai_pricing_configs.default_config.gross_margin → 兜底 0.30
- 建议规则：退货率 ≥15% / 净贡献 <0 → block；退货率 8-15% 或低规模 → watch；其他 keep
- 前端表格加两列 + 顶部 Alert 说明毛利率来源和"广告无地区拆分"免责

### 7.2 方案 3 调研：广告层能否按地区排除（`75a30ee` 文档）

Sub-agent 调研 WB/Ozon 官方文档 + 俄文卖家论坛，写入 `docs/api/region_sales.md §5`：

| 平台 | 效果广告 | 地区定向 | 地区排除 |
|---|---|---|---|
| WB | АРК / Auction | ❌ | ❌ |
| Ozon | Трафареты / Поиск | ❌ | ❌ |

**结论**：两平台效果广告**都做不到地区排除**。**Ozon Premium Plus 不值得升级**（只能事后复盘、不能前置阻断）。**唯一硬阻断：物流层关该地区配送**。

### 7.3 修退货率永远 0 的老 bug（`9f2e547`+`4baa6d8`+`cab5f56`）

用户追问"退货率没提取吧"——排查发现：

- **WB region-sale API 根本不返回退货字段**（实测只 9 个字段）
- 真实退货数据在 `/supplier/sales`：每条带 `regionName` + `saleID`（R 开头=退货）
- 新增 `WBClient.fetch_sales_returns_by_region`：按 (region, date) 聚合退货条数
- 修 `backfill_region_stats`：原版 31 天段汇总导致销售全堆到 date_to 一天，改为**按天逐日**循环
- 修 `_upsert_region_stats`：returns_map 里存在但 items 不覆盖的地区补建 `orders=0` 行（处理"当天只有退货没有新销售"场景）

**实测**：WB 近 30 天 1063 行 / 108 地区 / 1862 单 / 8 条退货（退货率 0.43%，真实数据就是这么低）。

### 7.4 地区 TOP SKU 决策视图 + 导出屏蔽清单（`cab5f56`）

**业务动机**：既然不能在广告层排除地区，唯一办法是"商品在该地区关配送" → 需要知道该地区**主要卖哪几个 SKU** 才能精准关。

- `WBClient.fetch_region_sales_by_sku`：按 region × nmID 双维度聚合
- `service.region_detail`：调 WB API + 反查 products.name_zh 返回 TOP N
- `GET /region-stats/region-detail` 新端点
- 前端地区行 `expandable`：展开显示该地区 TOP 10 SKU
- Card 右上角"导出屏蔽列表"按钮：CSV（UTF-8 BOM 兼容 Excel），自动筛 suggestion=block 的地区

### 7.5 Ozon 地区销售同步（`65043c8`）

- Ozon `analytics/data` **不支持** region dimension（实测 400 错误）
- 改用 **FBO posting 的 `analytics_data.city`**

## 九、追加体会

7. **平台 API 的"想当然"会坑你**：WB region-sale API 名字里有 "sale"，我本以为肯定返回 orders + returns，结果**只返回 orders**，退货要去 sales API（名字里没 region 那个）。API 文档命名规则≠字段完整性。以后遇到新平台 API 先拉一条真实数据看字段。

8. **"回填 = 按时间段一次拉"的陷阱**：原 backfill 31 天调一次 region-sale API，**API 返回的是 31 天汇总**（无 date 字段），导致销售全堆到 date_to 那天。按天循环虽然 N 倍请求但数据对齐。量级小（1 次请求/天）时请优先按天。

9. **"同名接口不同粒度"的隐性差异**：WB 和 Ozon 都有"地区销售"功能，但 WB 返回 regionName（联邦主体级）、Ozon 只能从 posting 拿 city（城市级）。两个数据并存时用户看表格会困惑（莫斯科 WB 几百单、Ozon 几十单）。代码注释明确标注了，但前端还没做差异化展示，下次有时间做。

10. **业务路径倒推的必要性**：用户原本问"能不能看地区 ROAS 决定不投广告"——我调研后发现广告层做不了，**但用户真正要的是"这个地区别花冤枉钱"**，就是砍"转化"不砍"支出"。于是做了物流侧屏蔽 SKU 的导出清单。核心是**不要被用户原始问法锁死**，挖到需求本质再选实现路径。

1. **"双 ID 字段"的必要性再证明**：昨天加 `platform_category_extra_id` 到 `category_platform_mappings`，觉得有点重，今天落地到 `platform_listings` 同一字段，让广告/属性接口 lookup type_id 不用每次即时反查。零解析开销。

3. **递归树遍历的"父向下传递"模式**：Ozon 分类树是"desc_cat 父 + type_id 叶子"两层标识，我最初按"同节点两值"写循环，错得离谱。正确模式是：父节点把自己的 id 作为 context 传给子节点，子节点叶子节点时用 (父 context, 自己 id) 组键。以后遇到类似嵌套结构（树形菜单、权限继承、分类路径）第一时间想 context 传递。

4. **实时计算字段的 Form.useWatch 胜过 setState**：以前做实时联动习惯 useState + onChange，现在发现 `Form.useWatch('cost_price', form)` 更直接 —— 不需要额外 state，不需要 onChange 同步，Form 内部一变就触发重渲染。代码量少一半。

5. **按钮禁用 + 二次确认 + message 反馈三件套**：批量复制成本是"看起来小动作实际会改多张表"的操作，用 `disabled={!hasCost}` 防误点、Modal.confirm 给挽回机会、message.success 展示结果数，三层防护让用户心里有底。
