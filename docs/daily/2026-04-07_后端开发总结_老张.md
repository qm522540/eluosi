基于架构师老林搭好的JWT认证框架（`dependencies.py` + `utils/security.py`），实现完整的认证业务流程。

#### 1.1 接口清单

| 接口 | 方法 | 认证 | 说明 |

#### 1.2 登录接口 `POST /api/v1/auth/login`

**请求体：**
```json
{
  "email": "user@example.com",
  "password": "password123"
}
```

**成功响应：**
```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "token_type": "bearer",
    "expires_in": 86400,
    "user": {
      "id": 1,
      "tenant_id": 1,
      "username": "张三",
      "email": "user@example.com",
      "role": "owner",
      "status": "active"
    }
  },
  "timestamp": 1775552000
}
```

**业务逻辑：**
1. 根据 email 查询 `users` 表，过滤 `status=active`
2. bcrypt 校验密码
3. 生成JWT（payload含 `user_id`、`tenant_id`、`role`），有效期1440分钟
4. 更新 `last_login_at` 字段
5. 失败返回错误码 `20001`（AUTH_FAILED）

#### 1.3 注册接口 `POST /api/v1/auth/register`

**请求体：**
```json
{
  "username": "张三",
  "email": "user@example.com",
  "password": "password123",
  "tenant_name": "XX贸易公司"
}
```

**成功响应：**
```json
{
  "code": 0,
  "msg": "注册成功",
  "data": {
    "user_id": 1,
    "tenant_id": 1,
    "username": "张三",
    "email": "user@example.com",
    "role": "owner"
  },
  "timestamp": 1775552000
}
```

**业务逻辑：**
1. 校验邮箱唯一性
2. 根据 `tenant_name` 自动生成 `slug`（正则替换非字母数字为`-`），重复时追加时间戳
3. 创建 `tenants` 记录（plan=free, max_shops=3）
4. 创建 `users` 记录（role=owner, 密码bcrypt加密）
5. 事务提交，失败自动回滚

```json
{
```
```json
  "name": "WB旗舰店",
```

```json
  "platform_seller_id": "123456",
  "api_key": "xxxxx",
  "api_secret": "xxxxx",
```

**业务逻辑：**

### 3. AI 服务客户端搭建

实现三个AI模型的统一调用框架，完善 `ai/router.py` 调度逻辑。

#### 3.1 架构设计

```
业务代码 (service层)
    ↓ 调用
ai/router.py (统一入口)
    ├── 任务类型 → 模型映射
    ├── 创建 ai_decision_logs 记录 (pending)
    ├── 选择客户端调用
    │   ├── deepseek.py → DeepSeek API
    │   ├── kimi.py     → Moonshot API
    │   └── glm.py      → 智谱 API
    └── 更新日志状态 (success/failed/timeout)
```

#### 3.2 基类设计 `services/ai/base.py`

`BaseAIClient` 抽象基类，统一接口：

```python
class BaseAIClient(ABC):
    async def chat(messages, temperature, max_tokens, system_prompt) -> dict
    # 返回: {content, prompt_tokens, completion_tokens, total_tokens, duration_ms, model}
```

**核心特性：**
- 使用 `httpx.AsyncClient` 异步HTTP调用
- 统一的超时处理和错误日志
- 自动计算调用耗时（duration_ms）
- 子类只需实现三个方法：`_get_headers()`、`_get_chat_url()`、`_parse_response()`

#### 3.3 三个模型客户端

| 客户端 | 文件 | 模型名 | 超时 | 用途 |
|--------|------|--------|------|------|
| `DeepSeekClient` | `deepseek.py` | deepseek-chat | 120s | 数据分析、ROI计算、广告优化、库存预测 |
| `KimiClient` | `kimi.py` | moonshot-v1-8k | 180s | 长文档处理、报告生成 |
| `GLMClient` | `glm.py` | glm-4-flash | 90s | 俄语文案、SEO标题优化 |

三个模型API均兼容OpenAI格式（`/chat/completions`），使用 Bearer Token 认证。

#### 3.4 路由调度逻辑 `services/ai/router.py`

**任务类型 → 模型映射表：**

| 任务类型 | 分发模型 | 说明 |
|---------|---------|------|
| `ad_optimization` | DeepSeek | 广告出价/预算优化 |
| `roi_analysis` | DeepSeek | ROI分析计算 |
| `inventory_forecast` | DeepSeek | 库存需求预测 |
| `report_generation` | Kimi | 日报/周报长文档 |
| `seo_generation` | GLM | 俄语SEO文案 |
| 未知类型 | DeepSeek | 默认兜底 |

**统一调用入口 `execute()` 函数：**

```python
async def execute(
    task_type: str,        # 任务类型
    input_data: dict,      # 输入数据（含messages或prompt字段）
    tenant_id: int,        # 租户ID
    db: Session,           # 数据库会话
    user_id: int = None,   # 触发用户
    triggered_by: str,     # 触发方式: manual/scheduled/alert
    system_prompt: str,    # 系统提示词
    temperature: float,    # 温度 (默认0.7)
    max_tokens: int,       # 最大输出token (默认2000)
) -> dict
```

**返回格式：**
```json
{
  "model": "deepseek",
  "content": "AI回复内容...",
  "tokens": {
    "prompt": 150,
    "completion": 300,
    "total": 450
  },
  "duration_ms": 2340,
  "decision_id": 1
}
```

**日志记录流程：**
1. 调用前：创建 `ai_decision_logs` 记录，状态 `pending`
2. 调用成功：更新为 `success`，记录 output_data、token消耗、耗时
3. 调用超时：更新为 `timeout`，记录错误信息
4. 调用失败：更新为 `failed`，记录错误信息（截断500字符）

**客户端实例管理：**
- 懒加载模式：首次调用时创建，后续复用
- 启动时校验API Key是否配置，未配置抛出 `ValueError`

**input_data 格式支持：**
- `{"messages": [...]}` — 直接传入OpenAI格式消息
- `{"prompt": "xxx"}` — 自动包装为user消息
- 其他dict — 自动JSON序列化为user消息内容

## 七、技术决策记录

### 决策1：注册时自动创建租户

**场景**：用户注册时是否需要先创建租户再邀请用户？  
**决策**：注册即创建租户 + owner用户，一个事务完成。  
**原因**：MVP阶段简化流程，后续可增加"邀请成员加入已有租户"功能。

### 决策2：AI客户端懒加载+单例

**场景**：三个AI客户端何时初始化？  
**决策**：首次调用时创建，存入 `_clients` 字典复用。  
**原因**：避免启动时因API Key未配置导致FastAPI启动失败；复用减少连接开销。

### 决策3：店铺删除使用软删除

**场景**：删除店铺是否物理删除？  
**决策**：软删除（status=deleted），数据保留。  
**原因**：店铺关联广告、商品、财务数据，物理删除会破坏数据完整性。已删除店铺从所有查询中排除。
