## 三、发现的真 bug（待修）

### 🔴 BUG #1：trend='new' 判定过宽（验证 04-19 §三十六 旧账）

**代码位置**：`app/services/bid/ai_pricing_executor.py:1679-1688`

```python
if p5["days"] == 0 and l5["days"] == 0:
    trend = "new"
elif p5["days"] < 3:         # ← 这一行是问题
    trend = "new"
elif l5["roas"] > p5["roas"] * 1.05:
    trend = "up"
elif l5["roas"] < p5["roas"] * 0.95:
    trend = "down"
else:
    trend = "stable"
```

**问题**：只要 `prev5` 窗口（5-10 天前那 5 天）有数据的天数 < 3，就判成 new，**忽略 last5 有多少数据**。

**触发场景**：SKU 曾停投几天，但后续连续投放。如：
- last5: 5 天全有数据
- prev5: 4 天停投（零花费不入库）+ 1 天有数据 = days=1 < 3
- **结果**：判 new —— 但 SKU 其实成熟了只是暂停过

**影响**：
- shop=6 实际：**49/80 SKU 被判 new**（61%）
- 这些 SKU 里 ROAS 健康的本应 small_up +5%，被打成 new 后走 no_change 跳过
- 决策器 `_profit_max_decision` 对 trend='new' 没单独分支（代码 `ratio>1.5 + trend==down → no_change` vs `!=down → small_up`），new 被当作非 down 但影响分支走向

**修复方案**（未动代码，待用户决策方案 A）：
```python
combined_days = p5["days"] + l5["days"]
if combined_days < 5:
    trend = "new"
else:
    # 正常算 up/down/stable，不因为 prev5 单窗口数据少就打回 new
```

合并判断：近 10 天加起来有 ≥ 5 天数据就不是 new，而非单看 prev5。

### 6.2 用户质疑 #2：如果每次 +3%，一直跌幅 <20%，会一直加价吗？

**关键发现：当前算法有复利爆炸隐患**。

代码路径：
```python
# ai_pricing_executor.py:1023
optimal_bid = int(round(current_bid * pm_decision["multiplier"]))
```

`current_bid` 每次从平台实时拉（即上次调完的新价）。所以 +5% 每小时一次会复利：
- 一天 24 次 × 1.05 ≈ 3.2 倍
- 一周 168 次 × 1.05 ≈ 爆炸

**现有防护**：只有单次幅度上限 `max_adjust_pct`（默认 30%），**没有日累计 / 偏离原价上限**。shop=6 因为 86% SKU 被判 no_change 未触发此坑。

### 6.3 BUG #3（用户发现）：样本量悬殊的 trend 比较不公平

用户举的场景：SKU 前 5 天投 5 天 ROAS=10，停 3 天，最近 2 天又投（l5.days=2, p5.days=5）。

代码 `_query_sku_history:1679-1688` 直接 l5.roas vs p5.roas 比较，**不看样本量差距** → 2 天数据可能因单次运气好坏剧烈波动，被误判 up/down。

shop=6 实证样例：
- camp 172 sku 379179926：l5.days=1, p5.days=4, 被判 stable（只 1 天数据无法判断）
- camp 181 sku 498605688：p5.roas=302.36 是单笔异常，噪声比信号大

**修复思路**：加档 `trend='insufficient'`（l5.days<3 或 p5.days<3），决策器走保守路径（亏就小降、不亏就观察）。

### 6.4 用户质疑 #3：为什么不是算出来一个值，而是按之前的数据？我们要利润最大化！

**这一问推翻了整个相对法架构**。

用户的两个洞察：
1. 目标是利润最高，不是 ROAS 最大
2. 每次都 current_bid × multiplier 是锚定原价，永远到不了理论最优

04-14 用户就提过增量 vs 绝对值之争（memory: `project_ai_pricing_algorithm_direction.md`），今日正式激活。

**利润最大化的数学本质**：
```
利润 = 收入 × 毛利率 - 花费 = (ROAS × 毛利率 - 1) × 花费
最大利润点 ≈ ROAS 略高于保本线（1/毛利率）的位置，让花费最大
```

**shop=6 实证（effective CPM 对比表）**：

| SKU | ROAS | 当前 CPM | 理论最优 | 晚8点应出 | 差距 |
|---|---|---|---|---|---|
| 185/531820756 | 22.42x | ₽35 | ₽219 | ₽263 | 高 533% |
| 168/372722952 | 19.84x | ₽49 | ₽228 | ₽274 | 高 369% |
| 176/260342372 | 13.36x | ₽29 | ₽149 | ₽179 | 高 406% |
| 184/499190031 | 4.51x | ₽35 | ₽55 | ₽66 | 高 58% |

**27/28 个成熟 SKU 当前出价严重不足，平均理论最优是当前的 3.67 倍**。

### 6.5 方案 D 诞生与被推翻

**方案 D（绝对值推导法）**：每次按公式 `target_cpa × CTR × CR × 1000` 算理论最优，朝它走半距。

**被用户推翻的原因（§6.6）**：公式假设 CTR/CR/aov 在调价后不变，但真实世界：
- CPM 从 35 涨到 219 → 展示量可能涨 2 倍、10 倍、没涨
- CTR 可能因流量质量变化跌 30%
- CR 可能崩盘
- aov 可能被稀释

**所以理论最优 ₽219 只是假设值，真实最优可能是 ₽80 或 ₽350**。甜点区**只能实测找，不能算**。

### 6.6 方案 E 诞生：爬山法（Hill Climbing / Gradient Ascent）

**核心思想**：不算理论值，每次小步试，看真实利润涨跌决定下一步。

**算法规则**：
- 冷启动按 ROAS 定方向：ROAS≥5x → +20% / ROAS 3.5-5x → +15% / ROAS 2.5-3.5x → +10% / 甜点区 → 不动 / 亏损 → -10~-20%
- 每 3 天评估一次利润变化（不每小时），定下个 3 天的方向
- 利润涨 → 继续同方向
- 利润跌 → 反向
- 步长动态收敛 20% → 10% → 5% → 2%
- 中间每小时跑 AI 时**只叠加时段/星期系数**，不改方向

**代码基础已有**：`_evaluate_profit_trial`（executor.py:312）就是爬山思想，但当前被埋在主决策器之后大部分 SKU 走不到。方案 E 把它提到主路径。

1. **"用户说'肯定要写啊' 推翻我的'明早做'"**

   17 条建议 vs §14.4 模拟的 18 条 —— 差 1 条被 ROAS 门控砍掉。这个 **1 条差距是系统级的可信度证明**：
