## 二、shop=6 AI 调价逻辑梳理（用户问题驱动）

用户报："shop=6 有 25 个活动计划但调价内容很少，怀疑被过滤太多"。

### 2.1 7 层过滤链路摸清

代码 `app/services/bid/ai_pricing_executor.py:773-1028`，从外到内：

| 层 | 条件 | 范围 |
|---|---|---|
| L1 | `shop_avg` 不存在（零广告历史）| 整店拒 |
| L2 | `shop_avg.cr == 0`（近 21 天无转化）| 整店拒 |
| L3 | 活动 `payment_type` 不在 {wb:cpm, ozon:cpc} | 整活动跳 |
| L4 | `fetch_campaign_products()` 失败/返空 | 整活动跳 |
| L5 | `current_bid <= 0` | 单 SKU 跳 |
| **L6** | `_profit_max_decision` 返 no_change/skip | **最致命** |
| L7 | `data_days ≤ 21` 不触发亏损检测 | 不出删除建议 |

### 2.2 L6 决策器 5 档（ratio = current_roas / breakeven_roas）

```
ratio < 0.5         → big_down   -30%
0.5 ≤ ratio < 1.0   → small_down -15%
1.0 ≤ ratio ≤ 1.5   → no_change  甜点区
ratio > 1.5 + down  → no_change  高位下行
ratio > 1.5 + up/stable → small_up +5%

data_days < 7 + roas=0    → skip 观察
data_days < 7 + roas≥保本 → no_change 数据少观察
```

### 2.4 "26 个活动只出 3 条建议" 的根本原因

1. 26 个 active 里 ~12 个活动 `data_days < 7`（断续投放 / 零花费日不入 ad_stats）→ 这些 SKU 全走"观察不动"分支
2. 店铺 ROAS 极端两极分化，没 SKU 落甜点区 → 大部分 SKU 按"数据少但已盈利"被判 no_change
3. 高 ROAS 区间里 trend='new' 判定过宽 → 49 SKU 被打成 new → 失去 up/stable 加价或 down 减价的机会

## 三、发现的真 bug（待修）

### 🔴 BUG #1：trend='new' 判定过宽（验证 04-19 §三十六 旧账）

**代码位置**：`app/services/bid/ai_pricing_executor.py:1679-1688`

```python
if p5["days"] == 0 and l5["days"] == 0:
    trend = "new"
elif p5["days"] < 3:         # ← 这一行是问题
    trend = "new"
elif l5["roas"] > p5["roas"] * 1.05:
    trend = "up"
elif l5["roas"] < p5["roas"] * 0.95:
    trend = "down"
else:
    trend = "stable"
```

**问题**：只要 `prev5` 窗口（5-10 天前那 5 天）有数据的天数 < 3，就判成 new，**忽略 last5 有多少数据**。

**触发场景**：SKU 曾停投几天，但后续连续投放。如：
- last5: 5 天全有数据
- prev5: 4 天停投（零花费不入库）+ 1 天有数据 = days=1 < 3
- **结果**：判 new —— 但 SKU 其实成熟了只是暂停过

**影响**：
- shop=6 实际：**49/80 SKU 被判 new**（61%）
- 这些 SKU 里 ROAS 健康的本应 small_up +5%，被打成 new 后走 no_change 跳过
- 决策器 `_profit_max_decision` 对 trend='new' 没单独分支（代码 `ratio>1.5 + trend==down → no_change` vs `!=down → small_up`），new 被当作非 down 但影响分支走向

**修复方案**（未动代码，待用户决策方案 A）：
```python
combined_days = p5["days"] + l5["days"]
if combined_days < 5:
    trend = "new"
else:
    # 正常算 up/down/stable，不因为 prev5 单窗口数据少就打回 new
```

合并判断：近 10 天加起来有 ≥ 5 天数据就不是 new，而非单看 prev5。

## 四、其他发现

### 4.1 shop=6 的 `is_active=0` 是用户自己关着的

Celery `run_bid_management` 每小时跑时**根本不会进到 shop=6**（代码 `bid_management.py:97-112`：查 time_active/ai_active 都 false 直接 return）。

用户想让 AI 每小时自动跑调价 → 要先在前端开启「启用 AI 调价」开关。这是配置问题不是 bug。

### 4.2 时段系数 + 星期系数已启用但"隐身"

`TIME_SLOT_MULTIPLIERS` + `DAY_OF_WEEK_MULTIPLIERS` 在 AI 调价 executor 里**写死启用**（executor:57-73, 882-884），只是设计裂痕：
- 主决策器 `_profit_max_decision` 只看 ROAS 比值决定"调不调"
- 时段系数只在"已决定要调"之后乘进 optimal_bid
- 大部分 SKU 被决策器判 no_change 跳过 → 时段系数没机会发挥 → 用户感知"时段没启用"

这是设计哲学问题，不是 bug。可能需要方案 C "AI+时段混合模式" 解决。
