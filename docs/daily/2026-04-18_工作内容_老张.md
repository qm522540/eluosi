## 三、新功能：搜索词洞察（Search Insights）

定位区分已有"关键词统计"：
- **关键词统计** = SEM 花费效率（我投的词花了多少钱）
- **搜索词洞察** = SEO 流量入口（用户实际搜什么找到我）

两者互补不替代。关键词统计改名决策**等老林**，本次菜单标签不动。

### 3.1 生产实测双重阻塞

用户确认要做之前，先 SSH 用 shop_id=1（WB-Shario）/shop_id=2（OZON-Shario）实测，都 403：

| 店铺 | 接口 | 返回 |
|---|---|---|
| WB shop=1 | `/search-report/product/search-texts` | `"Available only in a Jam subscription"` |
| Ozon shop=2 | `/v1/analytics/product-queries` 列表 | `"Analytics ... available starting from the premium subscription"` |
| Ozon shop=2 | `/v1/analytics/product-queries/details` | 同上 |

Ozon 试了 1 天/4 天/30 天三种窗口 + 两种 sort_by，全部 Premium 限制；参数格式坑：date_from 必须 RFC3339（`2026-04-03T00:00:00Z`），纯日期会 protobuf 报错；PageSize 上限是 100 不是 200；details 有独立必填 `limit_by_sku` ∈ (0, 15]。

### 3.2 用户决策：方案 B（做骨架，不做 Celery）

订阅开通后首次 refresh 再据实测字段微调代码。需要开通的订阅：
- **WB Jam** 约 ₽7,500-12,000/月（Стандарт 档即可，入口 seller.wildberries.ru）
- **Ozon Premium** 约 ₽2,990/月（普通档够用，入口 seller.ozon.ru/premium）

## 九、今日体会

1. **"能用的接口"和"当前账号能用的接口"是两回事** —— WB search-texts、Ozon product-queries 在
   官方文档里是"标准接口"，但实测才发现都要付费订阅。以后接新 API 先实测再动代码，
   不然文档读得再细也可能踩到订阅门。

2. **实测发现的小坑比文档还重要** —— Ozon date 必须 RFC3339（纯日期 protobuf 报错）、
   PageSize 上限 100 不是 200、details 有独立必填 `limit_by_sku`——这些官方文档都没明
   显标注，是靠 400 错误一条条试出来的。骨架代码里把这些边界条件都写死了，
   订阅到位后不用再重试 3 次才跑通。

3. **"订阅未开通 = 返回空"比 500 好得多** —— `SubscriptionRequiredError` + 93001 的设计
   让前端知道"这不是 bug 是缺订阅"，就能弹友好 Alert 而不是 `message.error` 轰炸。
   下次接任何需订阅的外部 API 都这么设计。

4. **"字段按推测写"也要标注明白** —— Ozon details 字段文档没给全，我按 Go client 源码
   推测写了代码注释里明确写 "基于社区 Go client 推测，首次上线需实测微调"，而不是
   装作很确定。以后用户开订阅后第一次跑可能要改，但至少 90% 骨架能复用。

5. **大功能也能小步推进** —— 从 0 到前后端完整骨架，拆成 6 个 commit 每步独立部署 + 验证。
   假如哪一步翻车（比如 service 层 SQL 错了），只需 revert 一个 commit 而不是整个功能。
   "小步提交"不是口号，是真省时间。
