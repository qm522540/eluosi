## 三、下午清 04-19 review 遗留 P1（4 commit）

### 3.1 P1#7 — `run_auto_exclude_now` 改 Celery delay 异步 `fa420b2`

**问题**：`app/api/v1/ads.py:1789-1825` 当前 `.apply().get(disable_sync_subtasks=False)` = 本地同步执行，hold FastAPI worker 几十秒（WB 串行 2s/req 累积）。

**修法**：
- 后端改 `.delay()` 返 `{task_id, msg}`
- 前端 `AdsOverview.jsx:407-424` message 改"任务已提交，10-30 秒后请查看日志"，不再等具体 `excluded` 数字
- `frontend/src/api/ads.js:245` 去掉 `timeout: 120000` 超长超时

**trade-off**：UX 从"点完立刻看到 X 个词被屏蔽"变成"等一会儿自己刷日志"。这是手动触发按钮，不是高频，体感可接受。

### 3.2 P1#10 — WB `set_excluded_keywords` 重试改精确匹配 `c606ff4`

**问题**：`app/services/platform/wb.py:779-782` 用 `w.strip().lower() == invalid_word.strip().lower()` 匹配被拒词。

**坑**：如果 current 里有大小写重复词（`["серьги", "Серьги"]`），WB 拒了 `"серьги"` → `strip().lower()` 两个都匹配 → `list.remove(target)` 只删第一个 → 真正被拒的留下 → 下次重试 WB 仍拒同一个词 → 消耗完 `max_retries=10`。

**修法**：`current.index(invalid_word)` 精确匹配 + `current.pop(idx)`。WB 错误信息里的词是原样回传，不需要 normalize。

### 3.3 P1#9 — `refresh_shop` 单商品级异常改 continue `05c6b46`

**老张 04-19 §七.G 的判断纠偏**：
- 原本 P2-7 说"SubscriptionRequiredError 整店 return 应改 continue"
- 老张澄清**不对**：Jam / Premium 都是店铺级订阅，一个 nm_id 403 后续 100% 也 403，continue 纯浪费 WB 3/min 限速配额
- **真正要改**：非订阅类的**单商品级**异常（429 限速 / 单 404 / network timeout）当前整函数崩，应 continue

**修法**：WB 循环 + Ozon 批循环各加 `except Exception` 分支：
```python
except SubscriptionRequiredError as e:
    return {...}  # 店铺级订阅，整店 return 保持不动
except Exception as e:
    errors.append({"nm_id": nm_id, "error": str(e)[:200]})
    logger.warning(...)
    continue  # 单商品级错误，跳过继续
```

之前定义但从未被用的 `errors = []` 终于启用了。

## 八、今日体会

1. **"工具选型"对架构师也是产品咨询**：今天上午 3 个问题全是"能不能 / 哪个好 / 要不要买"，不是写代码。但**回答精确度直接影响用户花不花冤枉钱** —— Seerfar 抓对价格 ¥4,790/年 vs MPStats ₽252,000/年 ≈ ¥21,000/年，价差 4 倍。用户这 1 万多人民币的决策我 30 分钟抓 6 个页面给出来，**这种时间投入产出比 = 开发 5 天**。

2. **"有接口让我系统调用吗"是技术合作最致命问题**：Seerfar 功能强、便宜、覆盖双平台 —— 看起来完美。一问有无 API 秒破功 —— 人工导出的半自动再好也架不住用户嫌麻烦。**"评估一个 SaaS 对我们有无价值，先看 API，再看功能"**。别被功能页面迷住眼。

3. **老张代码 review 的"G 项纠偏"是团队价值的体现**：我 04-19 P1 清单把"SubscriptionRequired 整店 return → continue"列为 P1，今天做的时候老张在 04-19 §七.G 早已判定**不改是对的**（订阅级错一个全整店错）。**他是代码作者，对细节的理解比 reviewer 更深**。我不盲目执行清单，读了 §七.G 后调整范围到"真正要改的单商品级异常 continue"。**清单是线索不是命令**。

4. **pre-commit hook 是低代价高收益的工程投入**：加这个撞号拦截总共 26 行 bash + 1 个看板文件。以后永远不会第三次撞号。老林 04-18+04-19 两次撞号的人工修正时间 ≈ 1 小时（改名 + 写历史说明 + 确认生产），未来一次撞号就抵消我今天 20 分钟的投入。**工程师花 20 分钟防 1 次事故是赚，团队规模越大回报越高**。

5. **"先等 Yandex 跑通再说吧"反映用户的产品节奏感**：当我一次性推 3 条方案（A 人工点击 / B 自动扫描 / C Chrome 插件）时，用户没纠结细节，直接判定"先 Yandex"。**用户在明确"主线 vs 副线"**——Yandex 是外部凭证阻塞，Seerfar 是内部决策。他知道主线一旦通了再看副线还需不需要。**工程师容易把全部方案当平权，但用户有优先级矩阵**。
