Explore agent 给了 7 个 issue，自己逐个核验，确认 **4 个真 bug**（已修并部署，commit `0d8f678`）：

### 2.2 致命 bug：`generated_history.py` 用 logger 但没 import

```python
# line 250
except Exception as e:
    logger.error(...)  # 但顶部从来没 from app.utils.logger import logger
```

**触发场景**：Ozon API 异常路径走到 except 分支 → `NameError: name 'logger' is not defined` → 用户看到 500 而不是友好错误信息。

昨天 `mark_title_applied` 改 async + 平台写回时落下的，except 分支没被自动测试覆盖到，**今天复盘才发现**。

### 2.3 `datetime.now(timezone.utc)` 写 DB 违反规则 6

```python
# line 197 (旧)
now_utc = datetime.now(timezone.utc)
```

**问题**：aware datetime 写 naive `DateTime` 列，与 CLAUDE.md 规则 6 字面对齐失败（"DB 写入唯一入口 `utc_now_naive()`"）。

修：`now_utc = utc_now_naive()`，from `app.utils.moscow_time`。

### 2.4 offer_id fallback 错（高风险）

```python
# 旧
offer_id = (offer_row.sku if offer_row else None) or row.platform_product_id
```

**问题**：sku 缺失时 fallback 到 `platform_product_id`（Ozon 内部 numeric ID），但 Ozon `attributes/update` API 需要 `offer_id`（卖家 SKU）—— **错配会调到错商品**。

修：fail safe，`offer_id = (offer_row.sku if offer_row else None) or None`，None 时上层走 skipped 提示，不会调错商品。

### 2.5 `ozon.update_product_attribute` 截断无告警

```python
# 旧
"values": [{"value": value[:max_chars]}]  # 静默截断，前端无感
```

**问题**：用户提交 800 字描述（max=500）会被截到 500，前端显示"已提交"以为全量同步成功。

修：加 `logger.warning("Ozon attr={} value 超长被截断 {} → {} 字符")` 让超长可被审计。

### 2.6 体会

最大教训：**异常分支不被自动测试覆盖时，里面的依赖（如 logger）极易漏 import**。考虑给 `tools/` 加个 lint：service 文件用了 `logger.` 但没 import 直接报错。

## 四、GLM vs DeepSeek 对比实测（中午）

用户：「我想加 DeepSeek 改动大吗」→ 答 2-3h（DeepSeek 客户端早就在 `app/services/ai/deepseek.py`，router 也已经在用，只 SEO 任务硬编码 `glm`）→ 用户：「拿 2 个商品生成给我看看，UI 后端都不改」

### 4.1 商品 1：OZON-E0149（耳环）

|模型|输出（post-check 后）|字符|耗时|
|---|---|---|---|
|GLM-4-flash|`серьги из медицинского сплава большие коричневые pt girl`|56|4.9s|
|DeepSeek|`серьги большие из медицинского сплава коричневые pt girl`|56|2.9s|

### 4.2 商品 2：OZON-N0002（项链）

|模型|输出|字符|耗时|
|---|---|---|---|
|GLM|`женская бижутерия на шею ожерелье подвеска с жемчугом колье на`|62|7.7s|
|DeepSeek|`ожерелье на шею женская бижутерия цепочка жемчуг подвеска с жемчугом кулон с колье на pt girl`|93|3.8s|

DeepSeek **词汇覆盖更全**（4 个项链同义词都进了），但**重复堆叠更明显**，post-check 后还有"кулон с"残段。

### 4.3 结论

| 维度 | GLM-4-flash | DeepSeek-chat |
|---|---|---|
| 俄语流畅度 | ✅ 略胜 | 接近 |
| 关键词覆盖 | 紧凑但偶有遗漏 | ✅ 更全 |
| 重复堆叠 | ✅ 更克制 | 偏多（依赖 dedupe 兜底）|
| 速度 | ~5-8s | ✅ ~3-4s（**快 2x**）|
| 成本 | ✅ glm-4-flash **免费** | 输出 ¥8/M tokens |

**结论给老板**：GLM 出来"完成度"更高开箱即用；DeepSeek 更快但要更厚的后处理保护层。**GLM 是免费 + 略稳，DeepSeek 没有压倒性优势**。是否暴露给用户**等老板拍**。

## 五、WB 搜索词同步改"按天补缺失"模式（下午）

用户准备测试搜索词洞察的 WB 同步，发现两个问题：

### 5.1 搞清楚"同步是同步几天"

我**先答错**为"锁死 7 天"——实际复核：
```js
// SearchInsights.jsx:89
await refreshShopSearchInsights(shopId, datePreset === '30d' ? 30 : 7)
```

页面顶部有"近 7 天 / 近 30 天"下拉，**默认 30 天**。我之前看 `api/search_insights.js` 函数签名 `days = 7` 默认值就以为锁死了，没看调用方实际传值。

### 5.2 用户：「WB 也能按天拉吧，和 Ozon 当时一样」

实测验证：WB `POST /api/v2/search-report/product/search-texts` 的 payload 是 `currentPeriod: {start, end}`，跨度 ≤31 天，**接受 start=end 单日窗口**。

代码里那段注释（"WB 仍 N 天聚合，API 限流没法按天拉，WB quota 恢复后再迁移"）写得偷懒——**不是不能拉，是怕烧光 quota**。

### 5.3 用户拍：「能实现就改，千万别重复」

shop=6 现有数据：
- 2 个 stat_date：4-22（85 行）+ 4-26（252 行）
- 都是 7 天聚合快照，跨日 SUM **会重叠**（4-15~4-22 双计）

### 5.4 实施（commit `37a3ed0`）

`refresh_shop` WB 分支重写为镜像 Ozon 的"按天补缺失"模式：

```python
# 1. 算窗口 [start_date, end_date]，end_date = today-2 (WB 没保护期)
# 2. 查 DB 已有 stat_date
existing_dates = {...}
# 3. force=True 清窗口内已有重拉，否则只补缺失
# 4. 计算 missing_dates
# 5. 按天循环 + 内层 nm_id 分批 (5 批 × 22s sleep)
for d in missing_dates:
    for batch in nm_id_batches:
        items = await wb.fetch_product_search_texts(
            nm_ids=batch,
            date_from=d, date_to=d,  # 单日窗口!
        )
        upsert(stat_date=d, ...)
```

**保留**：quota pre-check / WBSellerQuotaExhausted 中途熔断 / EMPTY_BATCH_ABORT 跨天累计早退。

**删**：原 WB 专用的 `_snapshot_exists` 快速 return 函数（按天模式不适用，孤儿清理 - 规则 3）。

### 5.5 数据迁移

旧 7 天聚合数据跟新按天数据**口径完全不同**（4M vs ~570K/天），混用会让跨日 SUM 数字乱。

直接删 shop=6 全部 337 行旧数据：
```sql
DELETE FROM product_search_queries WHERE shop_id=6 AND tenant_id=4 AND platform='wb';
-- 已删 337 行，shop=6 剩余 0 行
```

shop=1（WB-Shario）今早测试时**新发现**没开 Jam 订阅（403 "Available only in a Jam subscription"），memory 里"待确认"那条**今天确认**了。

### 5.6 成本

| 场景 | 调用次数 | 耗时 |
|---|---|---|
| 首次 30 天 | 100 SKU/20 = 5 批 × 30 天 = 150 次 | ~55 分钟 |
| 首次 7 天 | 35 次 | ~13 分钟 |
| 次日点 | 1 缺失天 × 5 批 | ~2 分钟 |
| 全有 | 0 次 | 秒返 |

⚠ **首次会很烧 quota**，建议测试时先选"近 7 天"验证 13 分钟跑完，再扩 30 天。

## 十七、今日体会(下午追加 — trip 事故痛感)

6. **"deploy 完叫用户刷新前端"是高风险 muscle memory** — 默认刷新页面是无副作用的,但 dashboard / 聚合页有外部 API 副作用(today-summary 调 WB)。规则修订:**deploy 完先自己 curl 验后端 + 看 Redis 状态,确认无外部调用副作用,再让用户刷新**。

7. **hook 设计前提的盲区:"主动同步" vs "被动浏览"** — 我设计 catalog 时按"功能 source"分(orders/keyword_stats),只想到主动调用入口(beat task / 手动同步按钮),完全漏掉**用户登录/刷新/查看页面时的被动 fetch**。这类调用往往以"实时聚合""快速预览"为名,实际也调 WB API 共享 quota。**未来加 hook 不能只看"调用名是不是 sync",要看"是不是真的发了 HTTP 请求到 WB"**。

8. **终极方案应该早做** — 老林给"WBClient._request 顶层拦截"建议的同时,我意识到这本应该是 day 1 设计 — 因为"开发记得给每个 endpoint 加 hook"是不可靠的人力假设,客户端拦截才是健壮设计。今天因为赶时间走"端点级 hook",留下覆盖度漏洞。**下次类似限流/quota 问题,先想客户端层拦截,而不是路由层逐个加**。
