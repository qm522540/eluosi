## 二、今晚踩坑与体会

### A. 服务器直接改代码是慢性毒药

小明这次犯的错：
1. 升级包的"替换孤儿组件"实际没接线（AdsAIPricing.jsx 没被任何 import），自己没测就直接上服务器
2. 服务器上 scp 文件绕过 git，本地/origin 不知道线上状态
3. 用户下午以为已经升级，实际看到的还是 v1 UI 几小时都没被发现

**防御**：
- CLAUDE.md 第一条"不要用阿里云 Workbench 手动 SSH 操作"要更严格地落到团队每个人
- PR 合并前必须本地 `npm run build` 验证组件接入关系（本地跑不起来就是孤儿）
- `deploy_remote.sh` 下次考虑加一步 `git diff HEAD origin/main --stat`，如果服务器工作区有未提交改动直接 abort 并告警

### B. git 冲突救火 SOP

今晚连续两次 deploy 因服务器 working tree 脏状态失败。救火 SOP 入记忆：

1. 先 `md5sum` 对比服务器文件 vs 本地 `new/` 或其他"本应同步过来"的来源
2. md5 一致 → 服务器内容没丢，`git checkout -- <file>` + `rm <untracked>` 清干净
3. md5 不一致 → 服务器有独占改动，先 `scp` 拉下来合入 git 再说
4. 然后才敢 `git pull`

### C. 前端 bundle hash 变化能快速定位"构建没生效"

今晚连续看 `index-CceMoRh2.js → index-CdvEPis5.js → index-BuB8Bob3.js → index-CTXTXPOJ.js`，每次 hash 变就说明 `npm run build` 真的跑出新产物了。用户说"还没生效"时，先看 hash 是否变，能快速排除"构建失败/缓存"等假象。
