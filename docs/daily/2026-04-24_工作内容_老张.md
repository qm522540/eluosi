## 二、用户报「OZON-pt-gril 没数据」— 三层诊断

### 2.1 现象

用户在 SEO 管理-优化建议页面选 OZON-Pt.Gril 显示「当前条件下无候选词」。

### 2.2 第一层：表层数据诊断

shop=7 OZON-Pt.Gril (tenant=4)：

| 表 | 行数 | 备注 |
|---|---|---|
| `platform_listings` | 372 ✓ | 商品同步正常 |
| `ad_campaigns` | 48 但**全 archived(42)+paused(6)，0 active** | |
| `ad_stats` | 262 行（最早 3-25） | 历史有数据，但 30 天内 0 行 |
| `product_search_queries` | 0 行 | Premium API 没拉到 |
| `seo_keyword_candidates` | **0 行 → rollup 显示空** | |

对比 OZON-Shario (shop=2)：候选池 3745 行 + ad_stats 382 行 + 318 orders ✓

### 2.3 第二层：用错指标判错根因

第一版分诊代码（`3e4d172`）用 `ad_stats` 判源数据 → Pt.Gril 有 262 行 → 返 `engine_not_run` → 让用户去手动跑引擎。

我直接生产试跑 `analyze_paid_to_organic(shop=7, days=30/60)` 全部 `analyzed_pairs=0`。

### 2.4 第三层：真根因 — Trafareti CPC 自动广告无关键词

读引擎 `self_sql` 发现 `JOIN ad_keywords kw` 是强依赖。查 `ad_keywords shop=7: 0 行` —— 这家店所有活动是 **Ozon Trafareti CPC（自动广告）**，没有关键词级数据，老张昨日 memory `project_ozon_keyword_stats_pending.md` 已有铺垫。

候选池引擎 `analyze_paid_to_organic.self_sql` 的真实链路：
```
ad_keywords → ad_groups → ad_campaigns → platform_listings → products
                      ↑ LEFT JOIN ad_stats（按 keyword_id 关联）
```

`ad_keywords=0` → 第一个 JOIN 就 0 行 → HAVING 也无数据 → 永远空。

**Pt.Gril SEO 优化建议页永远 not_ready 是业务正确**（不是 bug），新文案让用户知道是"广告类型选择问题 + Premium 订阅未同步"，不是系统坏了。

### 3.2 `8c336ac` `generate_daily_report` 时区 aware/naive 减法 bug

老林昨日 §17/§18 全仓扫尾时漏了 `app/tasks/report_tasks.py`。task_logs 04-22/23 每天 5:00/5:05/5:10 三轮全失败 `can't subtract offset-naive and offset-aware datetimes`，**已挂 4 天**。

根因：line 50 `started_at=datetime.now(timezone.utc)`(aware) → `db.commit() + db.refresh()` 后从 naive DateTime 列重读变 naive；line 101 `finished_at(aware) - started_at(naive)` 抛 TypeError。

修法（规则 6 对齐）：4 处 `datetime.now(timezone.utc)` → `utc_now_naive()`，naive-naive 等价 timedelta，且与 DB DateTime(无 tz) 列直接匹配。import 清理掉 `datetime/timezone`。

smoke：同步触发 `generate_daily_report.apply()` → SUCCESS ✓ task_log id=1340。

### 3.3 `b9750ef` 候选池分诊 v2 — 改用 ad_keywords + Trafareti CPC 文案

二次诊断后修正 v1 的判错：
- 分诊查询从 `ad_stats` 改为 `ad_keywords`（精准对应引擎依赖）

### 3.4 `6b1ee1d` `generate_daily_report` duration_ms 负数

8c336ac 修完后 smoke 出现 `duration_ms=-103`。根因：MySQL DATETIME(0) round 到秒（57.799 → 58），`db.refresh` 后 `task_log.started_at` 损失毫秒，`finished(57.902) - 58 = -98ms`。

修法：用 Python 局部 `started_at_local` 算 duration，不依赖 DB round 后的值；兜底 `max(0, ...)` 防其他罕见情况。

smoke：duration_ms=42 ✓ task_log id=1341。

### 3.5 `22d1ebb` `fetch_product_search_texts` 让 WBSellerQuotaExhausted 穿透

实测 04:00 MSK beat（实跑 01:53-55 UTC）：shop=5/6 cooldown 命中后还跑 3 批白等 40s。根因：`fetch_product_search_texts` line 1597 `except Exception` 把 `WBSellerQuotaExhausted` 也吞了（log warning + return []），上层 `refresh_shop` 的 `except WBSellerQuotaExhausted: break` 永远进不去。

修法：try 链最前面加 `except WBSellerQuotaExhausted: raise` 让异常穿透。

smoke：直接调 `refresh_shop(shop=5, days=7)` → `elapsed=0.02s` 第一批就 break ✓ errors 里规范记 `type=quota at_batch=1`（之前 40s+）。

## 七、今日体会

1. **"用错指标会让根因诊断转两圈"**
   v1 用 `ad_stats` 判源数据，结果 Pt.Gril 有 262 行历史 ad_stats 但 0 行 ad_keywords，
   分诊给出 `engine_not_run`，让用户去跑引擎 → 全空 → 再回来。**真正应该判的是引擎 SQL 强依赖的表**（这里是 ad_keywords），不是表面"广告统计"。
   下次写分诊判断前必须先读一遍引擎 SQL 的 JOIN 链，确认从哪个表"上游"开始决定结果。

2. **"全仓扫尾要扫所有目录，不是 service 优先"**
   老林昨日 §18 说"规则 2/6 全仓 0 残留"，但实际 `app/tasks/report_tasks.py` 漏了 —
   每天 daily report 三轮全挂已 4 天没人发现。**`grep -rn "datetime.utcnow\|datetime.now()" app/`
   要扫到 app/ 根，不能只扫 services/ + api/**。今天我修这个时也只查了 `report_tasks.py`，
   `app/tasks/` 其他文件可能还有同问题（待老林扫）。

3. **"`except Exception` 吞自定义异常是埋雷"**
   `fetch_product_search_texts` 内部 `except Exception` 兼容性写法，把老林新加的
   `WBSellerQuotaExhausted` 也吞了。这是经典"自定义异常被通用 catch 拦截"的反模式。
   **新加 raise xxxError 时要检查全调用链有没有 except Exception 把它吞了**，否则
   raise 不到上层等于没加。

4. **"日志里看到 cooldown 还能继续 sleep 是强信号"**
   shop=5 日志显示 batch 1/2/3 三连 quota cooldown，每批之间还有 20s sleep — 这种
   "明知道挂了还在等" 的模式 99% 是异常没穿透。读日志时遇到"重复失败 + 固定间隔"
   要立刻怀疑异常吞掉而不是 retry 策略。

5. **"用户单店报错暴露多层 bug"**
   用户一句"OZON-pt-gril 没数据" 暴露了：①hint 文案不准确（已修 v1→v2）；②老林全仓扫尾
   漏 report_tasks.py（已修）；③老林熔断到 _request 但 fetch_xxx 仍吞异常（已修）；
   ④WB 配额不是 daily reset（认知刷新）；⑤Trafareti CPC 自动广告无关键词数据（业务认知）。
