## 二、第一波：方案 E 上线后的实战 Bug 暴露（上午）

### 2.1 sku 498605688 的"利润涨却继续降价"

用户问 27 条建议为什么全部降价 -50%~-70%。深度分析后发现**多个真 bug**：

**Bug A：cold-start 被 prev5 噪声骗**
- prev5 实际只有 4-13 那天 spend ₽6.77 / 偶然 1 单 ₽2688 → ROAS 397x
- 算法以为"近 5 天 ROAS 3.4 / 上 5 天 ROAS 397，跌 99%" → 判 -5% 谨慎降价
- 但 397x 是噪声（一周几乎没投放）

**Bug B：日评估"过去 3 天 vs 再前 3 天" 系统性判涨**
- shop=6 投放断续场景下两边窗口都接近 0 → 微小绝对差判涨
- 27 SKU 中 25 个判涨，等于"无反馈自由落体"

**修法**：
- 引入"健康天清洗"概念
- cold-start 改用"剔噪声后凑齐 5+5 健康天" ROAS 比较（c7e30be / 900ed5f）
- 日评估改用"凑齐 3+3 健康天" 利润比较

**实测验证**：旧 25 涨/1 跌 → 新 12 涨/15 跌，分布回归合理。

### 2.2 sku 498605688 在 21 小时内被评估 5 次

实战发现 hill_base 从 ₽33 → ₽27 → ₽21 → ₽23 → ₽22，step 从 0.20 → 0.10 → 0.05。

**根因**：算法设计假设"base 一天动 1 次"，但代码每次 `_hill_climbing_decision` 都跑评估。Celery 每小时 + 用户每点 analyze_now 都会击穿。

**修法**：加"一天 1 次评估护栏"（7908bf8）。先用 23h 阈值，后改用**莫斯科自然日切换**（74335b8）—— 用户拍："时间点应该可预测，凌晨 0:05 固定"。

数据正确性等价（爬山法日评估按"日期"取数据），但语义更直观。

### 2.3 用户拍调时段系数

凌晨低谷 0.50 太狠，常触发平台最低价兜底。改：
- 0-4 时：0.50 → **0.70**
- 5-6 时：0.60 → **0.80**
- 23 时：0.65 → **0.80**
- 高峰段（10-13 / 19-22）保持不变（2fb59dd）

## 四、第三波：清洗规则的三轮迭代

实战发现"清洗"是个动态平衡，迭代了三版：

**v1**：剔 ROAS>50 OR spend<₽10  
- 救了 prev5=397x 噪声
- **但伤了长尾 SKU**（每天就 ₽5-10 投入）

**v2**（dd62620）：去掉 spend<10 阈值，只剔 ROAS>50  
- 长尾 SKU 不被一刀切
- **但还伤长尾 SKU 偶然出 1 单的天**（投 ₽7 出 1 单 ROAS 67x → 被剔）

**v3**（c061ae9）：双条件 **ROAS>50 AND spend<5** 才剔  
- 397x（spend 6.77）仍被挡 ✓
- 67x（spend 7.63）保留 ✓ 长尾真转化信号
- 长尾 SKU 每天 ₽5-10 数据全保留

**3 处函数同步改**（用户每次都强调"3 处都要同步"）：
- `_calc_healthy_window_metrics`（cold-start 5+5 / 日评估 3+3 用）
- `_calc_healthy_window_full`（reason recent_seg 用）
- `_get_roas_21_30`（亏损建议删除用）

## 六、生产事故 + 急救

### 6.1 SSE 流 nginx 60s upstream timeout

用户反馈"分析失败：network error"。nginx error log 实证：
```
13:09 upstream timed out (110) /analyze-stream
13:25 upstream timed out (110) /analyze-stream
```

`analyze_stream` 调 `_analyze_now_inner` 时 await 整个完成（27 SKU + WB API 60-90s），SSE 流期间无数据 → nginx 切断。

修：SSE 加 keepalive 心跳（11f8385）每 10s 发 SSE comment 行。

### 6.2 /suggestions 接口慢查询导致 nginx 499

用户反馈"历史建议列表显示不出来"。nginx 显示客户端 abort（499）—— 后端响应慢前端等不及。

诊断后发现 `LEFT JOIN ad_groups` 有索引但执行计划走偏。加 052 组合索引 `idx_camp_pgid (campaign_id, platform_group_id)` 修复。

**事后 review 发现**：052 加的索引跟已存在的 `uk_campaign_sku` UNIQUE 索引同字段冗余。无害但浪费存储，下次清理时删。

### 6.3 fastapi spawn error 端口冲突

每次 `bash deploy_remote.sh --backend` 后 fastapi 都 spawn error。原因：supervisor stop 旧 fastapi 时 worker 没及时释放端口 8000。手动 `fuser -k 8000/tcp` + restart 解决。

**遗留 P1**：supervisor 配置应加 `stopwaitsecs = 30` 或 `stopsignal = TERM` 让旧进程优雅退出。每次部署都手动 kill 太烦。

### 6.4 SSH 反复被远程关闭

下午 ~12:00 起 SSH 频繁被远程关闭，尤其 ALTER TABLE 命令。怀疑 sshd 限流或网络抖动。

最严重的：迁移 051 `ALTER ai_pricing_suggestions ADD recent_ctr/cr/profit` 反复跑 4 次都失败。**改方案绕过**：把数据塞 reason 字符串末尾，前端 regex parse（e571c04）。0 schema 变化解决问题。

教训：**SSH 不稳时 ALTER 必失败 → 优先选无迁移方案**。

## 十三、今日体会

1. **"用户实战反馈是最快的 bug 探针"**  
   今天发现的所有真 bug（SELECT 漏字段、CTR×100 重复、subject_name 是品类、sku 过滤漏掉、ON DUPLICATE 误诊…）**全是用户在测试中一条条问出来的**。我自己 self-review 永远找不出这些。**用户每"问一个为什么"都对应一条隐藏的算法或数据假设错位**。这种实战驱动的迭代密度比单纯的 code review 高 10 倍。

2. **"清洗规则三轮迭代 = 业务直觉的正确反馈"**  
   v1 一刀切 spend<10 → v2 只剔 ROAS>50 → v3 双条件。每一次都是用户用具体 SKU 把规则的"误伤面"打出来。**没有 sku 291661515（长尾）实例，我永远不会想到"投 ₽7 出 1 单 ROAS 67x 是真信号"**。算法规则的精度是被反例校准出来的，不是想象出来的。

3. **"Tooltip 信息密度决定算法可信度"**  
   旧静态 tip "ROAS 近期持续下滑且低于保本线" 让用户完全看不到判定细节。改成多行后用户能直接看到"近 5 天 CTR/CR/利润 + 21-30 天前 ROAS vs 保本"，**判断对错只需 3 秒**。**算法做得再对，UI 不透明用户也不会信**。今后所有触发"建议删除"/"大幅降价"的决策都该有这种四段式 Tooltip。

4. **"绕开比强攻更稳"**  
   迁移 051 反复 ALTER 失败时，我浪费了 30 分钟反复重试 SSH。最终决定**绕开** —— reason 字符串塞数据 0 schema 变化。**SSH/服务器有问题时最快的解法不是修 SSH，是设计一条不依赖 SSH 的路径**。这条思路能推广：任何"运维通道不可靠"的瞬间，都该问"能不能不走这条通道"。

5. **"Agent code review 误报是常态，但仍有价值"**  
   傍晚 agent 报"高危 #1 缺 UNIQUE KEY"，我查了 5 分钟才发现是误报。但**剩下的 4 个低危发现（死代码 / docstring 过时 / SSE keepalive 在 DB 阻塞期失效 / 052 索引冗余）每条都真**。Agent 的价值不在判断准，而在**强制系统性扫一遍盲区**。我自己今天写的代码自查根本想不到 SSE 在 DB 阻塞时心跳失效这种事。**Agent + 人工核验的组合是 review 的最优解**。

6. **"会话被关 + 网络抖动 + 服务器问题 三重压力下还能交付"**  
   今天会话被网络问题困扰多次（SSH 断连、Monitor timeout、迁移失败），但**所有真 bug 都修了**，最终用户验收的功能都跑通。关键在于：每次中断后都从 commit 历史 + nginx error log 立刻恢复上下文，不依赖会话内存。**日志 + 远程数据是最可靠的状态机**。
