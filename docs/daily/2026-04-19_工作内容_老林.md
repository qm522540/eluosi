## 九、今日体会

1. **没有版本表 ≠ 没有冲突**：`deploy_remote.sh --migrate` 这种"裸跑 SQL 按文件名"模式，团队规模一上来就是迁移撞号事故源。今天靠"两人各自占了 043"侥幸没出大事（生产两张都建了），下一次撞号可能是 `CREATE TABLE foo (a INT)` vs `CREATE TABLE foo (b VARCHAR)`，前者跑了后者就报"table exists"，后者那张表的字段永远进不了生产。**长期看应当加版本表 + 防重复跑机制**（比如 045+ 之后挂个 hash 校验）。

2. **架构师做 review 必查"两人改同领域"**：老林+老张都触 Ozon `product-queries` API、都建了底表，只看代码 review 不看全局功能图，就会错过这种数据冗余。**昨天的"明日对齐老张"不是客套，是真的得做** — 否则今天我们就有两个 Celery task 每天都跑同一份 API。

3. **小步提交真的能让收口工作有节奏**：今天 6 个 commit，每个独立部署可回退。中间死机一次（Edit 工具拒绝 git mv 后的文件），追加 commit 修头。没有任何一次"等到全部做完才推"，每步推完心理上才能继续下一个。

4. **P0 级"跨店聚合"bug 是规则 4 的标准案例**：用户场景永远是"先选店再操作"，但开发时容易写"按 tenant 全聚合"代码。这种 bug 不是逻辑错，是产品语义错 — 单元测试根本测不出来，必须靠 review。

5. **共用底表是设计层胜利**：老张的 `product_search_queries` 平台并集设计（加 `platform` 字段 + `extra` JSON）今天直接救场，让我半小时合并完成。**先建好基础数据资产再做上层应用** —— 04-17 的 `keyword_efficiency_rules` 也是同款，4 个上层功能直接复用没新建表。这是节奏的一部分。

## 三十、AI 调价改为利润最大化决策器 — `df46b01` + `f1015a3`

用户问："AI 该建议加价吗？我们目标是利润最大化吗"。我深入解释：
- 原算法是 **CPA-target**（让 CPA 趋近 target_cpa = max_cpa × 0.58）
- 但**利润最大化** ≠ CPA 达标 ≠ ROAS 最大
- 利润 = `(ROAS × net_margin - 1) × Spend`
- 利润最大点在 `ROAS ∈ [breakeven, breakeven × 1.5]` 区间（甜点）

**新决策器 `_profit_max_decision`**：
| ratio = current_roas / breakeven | 决策 |
|---|---|
| < 0.5 | big_down -30%（受 max_adjust_pct 上限） |
| < 1.0 | small_down -15%（亏损减耗） |
| 1.0 ~ 1.5 | **no_change**（甜点区不修没坏的）|
| > 1.5 + 趋势 up/stable | small_up +5%（试探多赚总额）|
| > 1.5 + 趋势 down | no_change（高位下行不冒险）|
| 数据 < 7 天 | 已盈利不动 / 亏损谨慎降 |

主循环 Step 1.5 先调用决策器：
- no_change/skip → 不出建议（continue）
- 其他 → optimal_bid = current_bid × multiplier

实测：用户 6 个 active SKU 全部判 no_change（甜点区）+ 有些 high 但下行不冒险。**算法的最高境界是知道什么时候不动**。

bug fix `f1015a3`：cfg.default_config 是 JSON 字符串，需 isinstance + json.loads。

## 三十五、AI 调价"利润最大化"算法策略详解（架构记录）

为什么换：
- 旧 CPA-target 让算法**追求 CPA 达标**而非利润最大化
- 用户 SKU 895084394 ROAS 4.26x（已盈利）被建议加价 100% → expected_roas 跌破保本 3.6x → 直接亏损
- 设计意图（项目记忆 `project_ai_pricing_strategy.md`）和实现矛盾

新算法核心：**利润 = (ROAS × net_margin - 1) × Spend**
- ROAS > breakeven (= 1/net_margin)：spend 越多利润越多
- 但有边际递减 → 利润最大点在 ROAS ∈ [breakeven, breakeven × 1.5] 甜点区

为什么不彻底重构而是包一层 hook：
- 保留 CPA 公式 + 所有保护层（max_cpa / 平台最低价 / ROAS 门控 / user_managed 锁）
- 在 Step 1.5 加入决策器，多数情况绕过 CPA 公式（直接 multiplier）
- 风险面小：决策器 bug 最坏只是不调价（保持原状）
- 一行 git revert 即可回滚

实测效果：用户的 6 个 active WB SKU 全部判 no_change 或 small probe，没有再出现 -94% / +100% 的离谱建议。

## 三十八、今日体会（再追加）

1. **"凉爽地承认假设错了"是高效修复的关键** — 我 04-18 commit 写"含空格短语全跳过是兜底"，今天用户截图明明该屏蔽却没列出，第一反应不是辩护是去翻 WB 文档。15 分钟确认假设错就改 3 层代码。**不要为旧 commit 辩护，那不是你的孩子**。

2. **算法的最高境界是知道什么时候不动** — 利润最大化决策器的 6 个 SKU 全部 no_change，乍看"算法没用"，实际是"系统在告诉你你出价都对"。这种"沉默"价值远超过频繁建议。**频繁调价 ≠ 智能 ≠ 有效**。

3. **WB 限速比文档严，需要实测找间隔** — 200ms 串行 50% 失败，2 秒才稳。WB 文档没说具体限速值，靠日志反推。**API 限速永远以实测为准**。

4. **数据库 schema 一个字段错位影响极大** — `ad_stats.ad_group_id` 实际存 WB nm_id 而非本地 ad_groups.id；`ai_pricing_suggestions.platform_sku_id` 实际存 Ozon 广告 SKU 而非货品 ID。**字段名 ≠ 实际语义**。修这类 bug 必须先用 SQL probe 出实际数据形态，再改代码。

5. **用户体验的"放不下"是 UI 设计师的盲区** — 我加 "商品ID xxx · 本地编码 xxx" 这个 label 完全没考虑表格列宽，用户截图我一看就明白。**任何加给表格内的文本都要按"假设列宽 80px"测一遍**。

6. **"自动识别"必须给"手动覆盖"兜底** — 净毛利率上传第一版我直接报错"未识别到列"，体验差。第二版改成返 headers + 5 行样本让用户手动选列。**自动化 + 手动 fallback** 是处理 messy 输入的标准模式。

7. **批量更新必须有"作用域"约束** — 用户警觉"没指定店铺会出错"。我立刻 SQL 验证 → 确实 10+ SKU 跨店重复。**用户的疑问就是 bug 报告，不要心存侥幸**。
