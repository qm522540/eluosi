## 九、今日体会

1. **没有版本表 ≠ 没有冲突**：`deploy_remote.sh --migrate` 这种"裸跑 SQL 按文件名"模式，团队规模一上来就是迁移撞号事故源。今天靠"两人各自占了 043"侥幸没出大事（生产两张都建了），下一次撞号可能是 `CREATE TABLE foo (a INT)` vs `CREATE TABLE foo (b VARCHAR)`，前者跑了后者就报"table exists"，后者那张表的字段永远进不了生产。**长期看应当加版本表 + 防重复跑机制**（比如 045+ 之后挂个 hash 校验）。

2. **架构师做 review 必查"两人改同领域"**：老林+老张都触 Ozon `product-queries` API、都建了底表，只看代码 review 不看全局功能图，就会错过这种数据冗余。**昨天的"明日对齐老张"不是客套，是真的得做** — 否则今天我们就有两个 Celery task 每天都跑同一份 API。

3. **小步提交真的能让收口工作有节奏**：今天 6 个 commit，每个独立部署可回退。中间死机一次（Edit 工具拒绝 git mv 后的文件），追加 commit 修头。没有任何一次"等到全部做完才推"，每步推完心理上才能继续下一个。

4. **P0 级"跨店聚合"bug 是规则 4 的标准案例**：用户场景永远是"先选店再操作"，但开发时容易写"按 tenant 全聚合"代码。这种 bug 不是逻辑错，是产品语义错 — 单元测试根本测不出来，必须靠 review。

5. **共用底表是设计层胜利**：老张的 `product_search_queries` 平台并集设计（加 `platform` 字段 + `extra` JSON）今天直接救场，让我半小时合并完成。**先建好基础数据资产再做上层应用** —— 04-17 的 `keyword_efficiency_rules` 也是同款，4 个上层功能直接复用没新建表。这是节奏的一部分。
