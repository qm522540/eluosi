| API | 端点 | 用途 |

**已实现方法：**

| 方法 | 功能 | WB API 接口 |
|------|------|------------|
| `test_connection()` | 连接测试 | `GET /adv/v1/promotion/count` |

### 2. 广告数据采集任务 (`app/tasks/ad_tasks.py`)

实现了三平台的广告数据定时采集任务，WB 为完整实现，Ozon/Yandex 为骨架。

**WB 采集任务 `fetch_wb_ad_stats`：**

```
触发频率: 每小时:05分 (celery beat)
↓
查询所有 active + 有api_key 的 WB 店铺
↓
遍历每个店铺:
  1. 同步广告活动列表 → ad_campaigns (upsert)
  2. 拉取最近2天统计 → ad_stats (upsert，按campaign+date+hour去重)
  3. 更新 shop.last_sync_at
↓
记录执行结果 → task_logs
```

**关键设计：**

- **upsert 策略**: 广告活动和统计数据都采用"存在则更新，不存在则插入"，避免重复数据
- **任务日志**: 每次执行记录到 `task_logs` 表，包含开始/结束时间、耗时、结果/错误
- **错误隔离**: 单个店铺失败不影响其他店铺采集，错误汇总到结果中
- **异步桥接**: 通过 `_run_async()` 在同步 Celery 任务中执行异步 WB 客户端调用
- **重试配置**: `max_retries=3`, `default_retry_delay=300`（5分钟后重试）

## 七、技术决策记录

### 决策1：Celery任务中使用 asyncio.new_event_loop()

**场景**: WB客户端是异步的(httpx.AsyncClient)，但Celery任务是同步执行的。  
**决策**: 每次任务执行时创建新的 event loop 运行异步代码。  
**原因**: Celery worker 默认不运行 asyncio event loop，使用 `_run_async()` 桥接最简单可靠。后续如需更高并发，可考虑使用 Celery 的 `gevent` pool。

### 决策2：广告统计拉取最近2天数据

**场景**: 每小时采集，应该拉取多长时间范围的数据？  
**决策**: 每次拉取昨天和今天的数据。  
**原因**: WB平台的统计数据可能有延迟，拉取2天范围确保数据完整性。通过 upsert 去重，不会产生重复。

### 决策3：ROI告警写入notifications而非直接推送

**场景**: ROI异常时是直接发企业微信还是先入库？  
**决策**: 先写入 notifications 表（channel=both），后续由通知服务统一推送。  
**原因**: 解耦告警检测和消息推送，通知服务(wechat_work.py)尚未实现，先确保数据不丢。
