## 三、silent-detector dryrun 观察 checklist（晚上 PT.Gril 30 天补拉用）

跑前：
- [ ] 确认 `wb:seller_quota_exhausted:*` Redis 0 key（trip cooldown 不在生效，确保观察的是 silent）
- [ ] 确认 PT.Gril 走的是 service.py 标准路径（refresh_shop），不是临时 ad-hoc 调用

跑中观察（合并日志 + DB + task_logs）：
| 信号 | 期望（通过） | 异常需回调 |
|---|---|---|
| API HTTP code | 200 全程 | 429 → 不是 silent，是 trip |
| 单天 inner loop 行为 | 连 ≥3 批 0 行 → 进 silent 模式（log: `silent rate-limit suspected`） | 进不去 silent 模式 → 假阴性，threshold/sleep 策略要查 |
| silent 模式 sleep | 30s → 120s 切换 | sleep 没拉长 → `pause_s` 选择逻辑 bug |
| silent 期间任意批返非空 | 退出 silent + log `silent mode exited` + reset consecutive_empty | 无法退出 → 状态机锁死 |
| 当天累计 silent 触发 | ≥5 → break 当天 + errors append `silent_rate_limit_day_abort` | 累计达 5 但没 break → 退出条件 bug |
| task_logs 记录 | status='partial'（refresh_shop 返 errors → record_sync_run 走 partial 路径）| status='success' → task 层 errs 兼容路径未走通 |
| 跨天兜底 | day_total=0 + EMPTY_DAYS_ABORT=3 整天 0 行 → 整体 early exit | 跑完 31 天没退出 → EMPTY_DAYS_ABORT 失效 |

判定：
- **通过**：上表全期望命中 + DB 至少新增 ≥1 天非 0 数据（说明真有数据被抓回，或者真的全 silent 时 detector 也正确退出）
- **回调阈值**：3 → 5（threshold 太激进。判定依据：PT.Gril 已知 4-21 / 4-22 / 4-23 是真稀疏天，如果 detector 在这些天误判为 silent 进入慢速 → 改 5）
- **假阴性**：HTTP 200 + 实际全 0 行 + detector 没进 silent → 检查 `consecutive_empty` 计数路径（可能 grouped 空时 day_total 累计不为 0 干扰判定）
- **副作用**：silent 模式下 sleep 120s × 31 批 ≈ 62 min，跑完时间是平时的 2x 多；如果运行超过 90 min 视为时间预算不足，要加 max_runtime 剪枝
