### 4.1 关键发现（按 §三 7 信号 checklist 逐条对账）

| 信号 | 实战结果 | 判定 |
|---|---|---|
| API HTTP code | **429 + 200 混合**（不是纯 silent） | ⚠️ 物种比预期复杂 |
| 单天 inner loop 连 ≥3 批 0 行 → 进 silent | ✓ batch 4 触发 | ✅ detector 正确触发 |
| silent 模式 sleep 20s → 120s | ✓ 切换工作 | ✅ 状态机 OK |
| silent 期间任意批返非空 → 退出 silent | 4-01 有 10 行入 DB → 退出过 silent | ✅ 退出机制工作 |
| 当天累计触发 ≥5 → break 当天 | 后续静默结束，没看到 break 日志 | ⚠️ 退出路径不清晰，要查 |
| task_logs partial 状态 | 搜索词洞察页面入口直连 fastapi，不写 task_logs（celery 路径才写）| 走错预期 |
| 跨天 EMPTY_DAYS_ABORT=3 整体 early exit | 没看到 "early exit" 日志，任务静默结束 | ⚠️ 可能正常退（per_day 写完）也可能异常吃掉 |

### 4.2 detector 的设计局限（重要）

**根本问题**：detector 的"连续 3 批 0 行"信号**没区分两种 0 行原因**：
1. **真 silent rate-limit**：HTTP 200 + 0 行 → sleep 长一点可能恢复 ✓ detector 设计意图
2. **429 后降级返 0 行**：HTTP 429 内置重试 3 次失败 → service 层拿到空 → 误判为 silent ✗ sleep 多长都没用

**实战撞的是混合物种**：WB 一边 429 一边静默 0 行，detector 的 sleep 120s 退避策略对 429 部分无效（WB 限流恢复时间 > 120s）。

**修法方向（不今天动，记账）**：
- 在 `WBClient.fetch_product_search_texts` 内部把 "429 重试 3 次都失败" 上报为更明确信号（如返特殊 sentinel 值或抛异常）
- service 层据此分流：true silent → 继续 detector / 429 burnout → 直接 quota_break 整体退出（避免浪费 sleep）

### 4.3 老张今日小结

- **✅ detector 代码功能正确**：触发条件、模式切换、状态机退出全工作
- **⚠️ 但物种识别有缺**：把 429 burnout 误判为 silent，给老林后续讨论
- **建议下次跑**：
  - 凌晨莫斯科时间（WB 低峰），减少 429 干扰
  - 跑前显式清 Redis 任何 cooldown key

落地约定（**关键：`tenant_id` 无 default 值**，让 IDE/lint 强制提示调用方显式传）：

```python
def some_helper(
    db: Session,
    tenant_id: Optional[int],   # 必填，但允许显式传 None；不写 = None 默认值
):
    """tenant_id 语义：
    - None:  Celery beat 全租户扫描（run_bid_management 等）— 规则 4 例外
    - <int>: 用户路径（CRUD / scan-now / enable / disable），
             必须传值做规则 1 纵深防御
    """
    q = db.query(Shop).filter(Shop.id == X.shop_id)
    if tenant_id is not None:
        q = q.filter(Shop.tenant_id == tenant_id)
    ...
```

调用 site 强制显式 kwarg（防位置参数眼花）：
- 用户路径：`some_helper(db, tenant_id=tenant_id)`

**为什么不写 `Optional[int] = None`**：有 default 值的话用户路径开发者会忘记传，纵深防御就失效。无 default = IDE 必报"missing argument"，强制看注释做选择。

- 后端 `ORDER BY revenue DESC, orders DESC` 取前 200
- 前端 Antd Table `defaultSortOrder='descend'` 按 `score` 二次排序

如果店里"score 高但 revenue=0"的词超过 200 条，后端会漏掉它们（按 revenue=0 时排序退化到 ID 自然顺序）。当前 ANI-OZON 1761 词里 revenue>0 不多，velu mode 能进 200 是因为 fallback。

修法两条路（不今天动）：
1. 后端默认 `ORDER BY` 改成 score（要把 score 算进 SQL，不是 Python 后排序）
2. 前端切 score 排序时同步刷后端 `sort=score_desc`（要后端支持这个 sort 参数）
