## 四、主线：全局映射 hints 飞轮（重点）

用户问"品类/属性映射是不是做成全局的"，深入聊后得出三层架构：
1. **纯翻译** → 全局（已做，A+B）
2. **平台 schema 经验建议** → 全局（今天做的）
3. **本地分类树 + 租户决策** → 租户级（不动）

### 数据表（037、038 迁移）

| 表 | 作用 |
|---|---|
| `global_category_hints` | 单平台分类"大家怎么叫"（lossy 投票，top1） |
| `global_cross_platform_category_hints` | 跨平台共现 `(platform_a, cat_a, platform_b, cat_b)`，按字典序规范化 |
| `global_attribute_hints` | 单平台属性"大家怎么叫" |

全部不带 `tenant_id`，跨租户共享。038 是补 `created_at` 字段（ORM `BaseMixin` 需要，037 漏了）。

### 读/写（P1 + P2）

**服务** `app/services/global_hints/service.py`：
- `record_category_confirmation(db, tenant_id, mapping)`：确认时调。写单平台 hint + 扫同本地分类的其他已确认映射，两两写跨平台共现
- `record_attribute_confirmation`：同上（只写属性 hint，不做跨平台）
- `get_category_hint / _bulk` / `get_cross_platform_hint` / `get_attribute_hint / _bulk`

**挂载点**：
- `confirm_category_mapping` / `confirm_attribute_mapping` 成功后触发写入
- `suggest_category_mappings`：本地分类已有其他平台映射时，**先查跨平台 hint**，命中跳 AI
- `suggest_attribute_mappings`：先批量查属性 hint，命中的直接用，只对 miss 集合调 Kimi

### P3 前端徽章

- `CategoryMappingTab` / `AttributeMappingTab` 加 ⭐N Tag + Tooltip "全网 N 个租户确认过此映射"
- `AISuggestCategoryButton` 结果区分 `source='global_hint'` 显示 "⭐全局(N 次确认)"，额外 info message 提示"省 token"

### P4 回填（冷启动种子）

`scripts/backfill_global_hints.py`：遍历所有 `is_confirmed=1` 的 CategoryPlatformMapping / AttributeMapping，调 record_* 批量灌入 hints。**不幂等**（重跑会叠加）。

生产回填后：
```
global_category_hints                : 4 行
global_cross_platform_category_hints : 0 行（生产数据里还没有同一本地分类两边都确认的组合）
global_attribute_hints               : 30 行（76 条属性映射去重后）
```

**跨平台 0 行的原因**：生产 tenant 4 里 "节日套装" WB 716 已确认 + OZON 17028733 **待确认**；气球/雨滴只 WB；时尚饰品只 OZON。点一下 OZON 那条的确认就会生成第一条跨平台共现。

### init-from-* 的 hint 贡献

早期实现里 `init-from-wb` / `init-from-ozon` 的 1:1 自动确认不经过 `confirm_*` 接口，绕过了 hints。这是最大的种子数据漏洞。

改法（`2336d7a`）：init 函数内收集自动确认的 mapping id 到临时列表 `_confirmed_cat_mps` / `_confirmed_attr_mps`，末尾统一调 `_contribute_hints` 批量写入。返回 stats 加 `hint_contributions` 计数。
