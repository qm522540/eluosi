### 5.2 列名"曝光/频次"含义混淆（`d5efaf3` / `274a500`）

用户：`фотозона из шаров` 在搜索词洞察里曝光 3518，但优化建议显示 21——为什么差这么多？

诊断：
- 搜索词洞察"曝光/频次"列实际取 `SUM(frequency)`
- 优化建议候选池 `organic_impressions` 取 `SUM(impressions)`
- 后端 SQL 跨 SKU 直接 SUM `frequency`，但 frequency 可能是 query-level 全网量按 SKU 复制——会重复算 N 倍

我把列名改成"搜索量"+"曝光"两列，SQL 改子查询 MAX(frequency) per (query,day) dedupe。

### 5.3 自承"对 frequency 字段假设错"（`e5e2b2f` 回滚）

用户："修改后数字反而更多了，1726"。

我重新查 04-23 数据：4 个 SKU `frequency` 完全不同（1073/831/80/61）——证明 **frequency 是 SKU-level 字段（Ozon `unique_search_users` = "看到该 SKU 的搜词独立用户数"）**，不是 query-level 复制。我之前只看 04-17/04-18 的两 SKU 巧合相同就拍板，是粗心。

立刻回滚 SQL 改回原 SUM，前端列名"搜索量"→"搜索次数"，Tooltip 重写为正确语义（frequency ≥ impressions = 被检索 ≥ 被滚动看到）。**用户视角数字回到原值 3351**（我之前手算少了 100，真实是 3251 还是 3351？后查得 3351）。

**教训**：聚合 bug 诊断必须扫多天多 SKU 数据，2 行巧合相同样本不足以判定字段是 query-level 还是 SKU-level。

### 5.4 加曝光比例列（`833dd07`）

用户问"曝光比例越高代表词越适合店铺吧"——部分对。加曝光比例列（`impressions/frequency`），≥60 绿 / 30~60 橙 / <30 红。Tooltip 提醒"相关性强 ≠ 适合卖"。
