## 三、老张同步 + `8c2f22c` BaseException 大招

老张早晨独立 review wb.py 发现 17 个 fetch_xxx 普遍写 `except Exception: return []`，会**吞掉** WBSellerQuotaExhausted，让 refresh_shop 顶层 `except WBSellerQuotaExhausted: break` 永远进不去。他在 `22d1ebb` 给 `fetch_product_search_texts` 显式补了 `except WBSellerQuotaExhausted: raise`，但要扫尾剩余 16 处。

我没在 16 个地方分别补 raise，而是**一行根治**：

```python
class WBSellerQuotaExhausted(BaseException):
    """asyncio.CancelledError 同思路 — 不是业务错误，是流控信号"""
    pass
```

继承 BaseException 而不是 Exception，`except Exception` 自动放行，无需在每个 fetch_xxx 加 raise。显式 catch by class（`except WBSellerQuotaExhausted as e:`）不受影响。Smoke MRO `[WBSellerQuotaExhausted, BaseException, object]` ✓。

**同 commit `8c2f22c` 还修了**：search_insights/service.py:355 pre-check 还在用 `quota_circuit_key_for_shop`，我 ab895eb trip 写的是 seller_ key 不命中。改成 seller_ 优先 + shop_ fallback。

## 四、`5408b07` SSE handler regression 修复

用户报"分析失败：network error shop=6"。诊断：

`nginx error.log`：
```
upstream prematurely closed connection while reading
GET /api/v1/bid-management/ai-pricing/6/analyze-stream
```

**root cause 是我刚 8c2f22c 引入的 regression**：BaseException 让 `except Exception` 自动放行，但 `analyze_stream` SSE handler 顶层 `except Exception` 也抓不到了 → 异常穿透 SSE generator → fastapi 强行关闭连接 → nginx upstream prematurely closed → 前端看到 "network error"。

修：在 analyze_stream 加显式 `except WBSellerQuotaExhausted as e:` 转 SSE error 消息 + `except BaseException` 兜底（CancelledError 必须 re-raise 让 asyncio 知道）。

**教训**：BaseException 让中间层 fetch_xxx 自动放行是对的，但**业务边界**（API handler / SSE generator / Celery task 顶层）必须显式 catch 转结果，不能让它穿透到 framework 层。这条加进 memory `reference_wb_seller_quota_per_endpoint_group.md` 第 8 条。
