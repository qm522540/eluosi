### Step 2/7 — service 引擎 `a722f30`

**两维度聚合**：
- **scope='self'**：商品自己的付费词 JOIN 链 `ad_keywords→ad_groups.listing_id→platform_listings→products`，HAVING `orders≥1 AND roas≥阈值`
- **scope='category'**：同 `local_category_id` ≥3 个商品共享的高 ROAS 词，扩散到类目全部商品

**覆盖判断**：`keyword.lower() in (title_ru+variant_attrs CAST AS CHAR).lower()` → 完全覆盖的跳过（不是反哺候选）。

**综合得分**：`来源数×2 + ROAS + log10(订单+1)×2`，上限 100。多源命中自动加权。

**规则合规**：
- 规则 1：所有 SQL `WHERE tenant_id = :tid` + JOIN 条件 `AND xx.tenant_id = :tid` + upsert `SET tenant_id = VALUES(tenant_id)`
- 规则 2：`datetime.now(timezone.utc)`
- 规则 4：service 接 shop 对象（路由层 get_owned_shop 守卫），全部 `WHERE c.shop_id = :sid`

4 个核心函数：`analyze_paid_to_organic / list_candidates / adopt_candidate / ignore_candidates`。

## 九、今日体会

1. **不要过度设计，但要为未来预留结构**：`seo_keyword_candidates` 表的 `sources` JSON 数组是关键 —— 一期只塞 `[{type:paid, scope:self}]`，未来 B/C/D 四个源追加都不改表，只 UPDATE JSON。**Schema 一次定好 vs 每次加源改表** 是长期节奏差 10 倍。

2. **"拆子组件"不是洁癖是真省事**：前端 560 行分 6 个文件，最大 190 行。老林 `KeywordStats.jsx` 877 行是反面教材。以后所有新模块**首个 commit 就拆子组件**，等到 500 行再拆重构成本至少 3 倍。

3. **working tree 并发的诡异陷阱**：453de8d 那个 commit 夹带我的未完成代码 —— 大概率是 Edit 工具、linter、pre-commit hook、git add 之间的 timing 问题。**教训**：多文件修改时， `git status -s` + `git diff --stat` 每一步查，**别信 Edit 的"文件已修改"自诊断，要靠 git 自己说**。

4. **5 源信号图是思考框架不是必做清单**：最初我只设计单源（付费）反哺，被用户追问"还有别的参考吗"才展开到 5 源。**好的架构师要主动把思考半径开足**，但**好的工程师要务实收缩到 MVP** —— 所以一期真实做的只有源 A + C1-a（1.5 源），其他 4 源架构预留不落地。

5. **订阅阻塞催生架构红利**：因为 B/D 源都要等订阅，被迫把源 A 单独做成 MVP —— 结果发现付费反哺自己就是一个完整功能，不依赖订阅也能交付价值。**外部限制有时候反而是架构收敛的驱动力**。

6. **昨日"纠错"今日翻盘**：我昨天的"搜索词洞察骨架 + 字段按推测写"被老林 04-19 review 出几个 P0，今天被他帮我修掉，腾出时间我做新功能。小步提交 + 小范围 review 的飞轮完美跑起来。如果昨天攒一大块一起推，今天就是在修 P0 而不是做新功能。
