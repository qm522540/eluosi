## 九、今日体会

1. **不要过度设计，但要为未来预留结构**：`seo_keyword_candidates` 表的 `sources` JSON 数组是关键 —— 一期只塞 `[{type:paid, scope:self}]`，未来 B/C/D 四个源追加都不改表，只 UPDATE JSON。**Schema 一次定好 vs 每次加源改表** 是长期节奏差 10 倍。

## 十六、收工后追加体会

1. **"字段按推测写"是纸老虎 —— 一跑真数据就原形毕露**
   上午设计 SEO 引擎、搜索词洞察骨架时都留了"字段靠推测，首次实测微调"的技术债。今天下午真数据一来就集中爆发：
   - `items` vs `queries` —— 一个字母差，全链路空
   - `orders_count` vs `order_count` —— 单复数差，订单永远 0
   - 传入 SKU vs 返回 SKU 不同 ID 空间 —— 映射全丢
   **教训**：下次接新 API，**骨架阶段就用 curl probe 一次真 response 粘在 docstring 里**，别等"有订阅了再说"—— 骨架代码 70% 可能是错的，实测时间比写代码还久。

2. **"数据源错了"比"算法错了"更隐蔽**
   上午 SEO 引擎 SQL 写得干干净净，规则 1/2/3/4 全合规，self-review 挑出 3 个 P0 但都是"聚合细节错"不是"根源错"。真正的根源是 **假设了一张实际没数据的表**。算法细节错 review 能抓，数据源假设错只有真数据跑一次才知道。**以后新模块第一步：先 `SELECT COUNT(*)` 验每张数据源表的生产数据量**，再动笔写 service。

3. **"主数据源翻转"比"加源"更直接**
   原本想"等 WB Jam / Ozon Premium / 源 C Wordstat / 源 D 爬虫四源齐全再升级" —— 但实际产品只需要 **product_search_queries 一路通** 就能跑起来。**多源设计是防御，单源跑通是进攻**。用户只关心"有没有数据能用"，不关心"接了几个源"。下次设计避免"等多源才能启动"的囚徒困境。

4. **Ozon `queries` 里零订单但高曝光的词 = 纯金 SEO 机会**
   Top 机会词里订单=0 但曝光 100+ 的词 —— 意味着用户搜了、系统展示了我商品、但我标题里一个字都没提这词 —— 改个标题就吃免费流量。这不是我设计时预料到的副产物 —— **好数据会告诉你好产品该怎么做**。

5. **"0 条 → 798 条 → 2909 条"的连锁反应**
   单点修复（Ozon 字段 bug 一个 commit 798 条）→ 架构性重组（SEO 接入源 B 一个 commit 2909 条）。整个链条下游的多个模块（SEO 引擎 / SEO 健康分 / AI 标题优化）全部等同一个上游数据源。**修好一个上游 = 解锁下游 N 个功能**。排查底层 API 字段 30 分钟的收益 = 下游一整条产品线的启动。
