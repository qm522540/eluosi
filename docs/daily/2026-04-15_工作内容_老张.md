### 6. WB 商品同步字段映射修正

patch.py 里的 `_sync_wb_products` 假设 WBClient 有 `fetch_all_products()` 方法且返回 `subjectName/salePriceU/quantityFull` —— 这是老 API 的字段，**全部错了**。

实测 `WBClient.fetch_products()` 调的是 `/content/v2/get/cards/list`，真实字段：

| 字段 | 映射到 |
|---|---|
| `nmID` | `platform_product_id` |
| `title` | `title_ru` |
| `description` | `description_ru` |
| `sizes[0].price` | `price` |
| `photos[0].big` | `products.image_url` |
| `vendorCode` | `products.sku` |
| `brand` | `products.brand` |

修完后 WB 店铺同步跑通。

### 7. Ozon 端点升级到 v3

Ozon 下线了 `/v2/product/list`（返回 404）。`app/services/platform/ozon.py` 里的 `fetch_products` 还在用 v2。

- `fetch_products`：改 `POST /v3/product/list`，支持 `last_id` 游标分页（不再用 page）
- **新增** `fetch_product_info(product_ids)`：调 `POST /v3/product/info/list`，批 1000 拿详情（list 接口只返回 `product_id/offer_id/has_fbo_stocks/archived`，真实字段 `name/price/old_price/images/barcodes` 都在 info 里）

**`_sync_ozon_products` 两步拉取流程**：

```
1. fetch_products(last_id="", limit=1000) → 拿 product_id 列表 + archived_map + stock_map
   ↓ 游标分页，最多 50 页（5 万商品封顶）
2. fetch_product_info(chunk=200) → 拿 name/price/old_price/images/barcodes
3. upsert PlatformListing
```

状态推断：
- `is_archived=true` → `deleted`
- `has_fbo_stocks + has_fbs_stocks` 都 false → `out_of_stock`
- 否则 → `active`

折扣价：`old_price > price` 时，`old_price → price` 字段（原价），`price → discount_price`（折后价）。

## 五、今日体会

- **patch.py 这种交付模式要小心**：它假设 service 层已有某些方法/字段，实际跑起来才发现 `fetch_all_products` 根本不存在、字段名全错。手动执行 5 个操作都成功 ≠ 功能能用，真正跑 Celery 任务才暴露问题。下次收到 patch 脚本要先过一遍引用的下游函数是否存在。
- **DB schema 和 ORM 必须同步改**：026 迁移在服务器上跑过了，但 `app/models/product.py` 没跟上，SQLAlchemy kwargs 报错。下次加字段要一起改三件：SQL / ORM / Pydantic。
- **Ozon API 版本迭代快**：一年时间 `/v2/product/list` 直接 404，没有 deprecation 期。以后接新端点时最好看下 Ozon 公开文档的"停用"列表。
- **不造占位任务更干净**：spread_products_task 本想加占位 celery task，但既然 API 端点能直接返回假结果，就根本不需要在 tasks 文件里留死代码。
