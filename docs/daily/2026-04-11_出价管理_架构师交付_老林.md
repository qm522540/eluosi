#### 3.2.2 `app/services/bid/time_pricing_executor.py`（新建目录+文件）

实现分时调价全部业务逻辑。要点：
- `enable(db, shop_id)` 必须用 `SELECT ... FOR UPDATE` 锁住 `time_pricing_rules` 和 `ai_pricing_configs`，校验互斥（规范 §9.4 有示例）
- `execute(db, shop)` 由 Celery 调用：
  - 跳过 `ad_groups.user_managed=true`
  - 对每个活跃 ad_group：按当前 period 计算 `new_bid = original_bid * ratio / 100`
  - 首次启用后 Celery 第一次执行时，把当前出价写入 `original_bid`
  - 调 Ozon 改价 API → 写 `bid_adjustment_logs`（`execute_type='time_pricing'`）→ 更新 `last_auto_bid`
  - 失败写 `last_execute_status='failed' + last_error_msg + retry_at=now+30min`
- `restore_sku(db, shop_id, sku)` 把出价改回 `original_bid`

#### 3.2.3 `app/services/bid/ai_pricing_executor.py`（新建）

AI调价执行器。要点：
- `enable(db, shop_id)` 同样用 FOR UPDATE 校验互斥 + 校验 `shop_data_init_status.is_initialized=true`
- `analyze_now(db, shop_id, campaign_ids=None)`：
  - 读取 `ai_pricing_configs` 当前 `template_name` 指向的 JSON 配置
  - 调 DeepSeek → 生成建议 → 写 `ai_pricing_suggestions`
  - 若 `auto_execute=true`：立即执行每条建议，跳过 `user_managed=true` 的 group
- `approve_suggestion(db, sug_id)` / `approve_batch(db, ids)`：
  - 必须校验 `generated_at` 是否为莫斯科日期的今天（过期返 `92005`）

## 五、架构决策说明（ADR摘要）

### 决策1：DROP 旧 ai_pricing_configs 重建而非 ALTER

旧表是品类级多行，新方案是店铺级单行+3模板JSON。结构差异过大，ALTER 无法平滑过渡。昨日日报显示 AI调价模块才上线一天，数据价值低。直接 DROP 重建最干净。

### 决策2：Celery 改为每小时整点触发（minute=5）

旧方案是每10分钟Beat检查 + 时段策略二次判断（复杂）。新方案统一为每小时05分触发，由执行器内部读取当前店铺规则决定行为（简单）。代价：不支持30分钟粒度的高峰期巡检，但业务上足够。

### 决策3：分时调价和AI调价互斥

两者都是"自动改出价"，同时开会产生叠加效果难以预测。强制互斥，UI也强制切换，通过 `SELECT ... FOR UPDATE` 保证原子性。

### 决策4：bid_adjustment_logs 和现有 ad_bid_logs 并存

`ad_bid_logs`（014）是给"自动化规则"用的广告级出价变更记录；新 `bid_adjustment_logs` 是"出价管理模块"的统一日志。字段结构差异大，不合并。未来若要统一，单独做一次日志表整合迁移。

### 决策5：ad_campaigns 清理 4 个 pricing_* 字段

新方案不再有"单活动覆盖模板"的概念。直接 DROP 减少 schema 噪音。若未来需要单活动覆盖，走 `ai_pricing_configs` 的 JSON 扩展方案。
