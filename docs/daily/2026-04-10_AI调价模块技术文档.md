## 一、模块概述

AI智能调价模块是本系统的核心功能，通过调用DeepSeek大模型分析广告数据，自动生成出价调整建议，支持Ozon自动执行和WB建议模式两种运营方式。

### 核心能力

| 能力 | 说明 |
|------|------|
| DeepSeek AI分析 | 调用DeepSeek-Chat模型，基于多维数据生成调价建议 |
| 五时段差异化 | 根据莫斯科时间自动切换高峰/平稳/低谷策略 |
| 三层数据融合 | 活动历史(60%) + 店铺基准(30%) + 实时数据 |
| 商品生命周期 | CTR+CR组合判断冷启动/测试/放量/衰退四阶段 |
| 大促周期识别 | 日历驱动预热/冲刺/恢复三阶段，自动调整出价系数 |
| 动态数据窗口 | 7-21天自适应窗口，排除大促异常数据 |
| 冷启动降级 | 新活动数据不足时以店铺基准为参考，调幅≤10% |
| 20分钟冷却 | Redis记录调价时间，防止同一活动频繁震荡 |
| 策略模板 | 标准/保守/激进三模板，支持单活动参数覆盖 |
| 双平台支持 | Ozon自动执行 + WB建议模式（企微通知+后台直链）|

### 2.2 调用链路

```
Celery Beat (每10分钟)
  └→ ai_pricing_task.check_and_run_ai_pricing()
      ├→ 读莫斯科时间 → 匹配时段策略(time_strategy.py)
      │
      ├→ 遍历 Ozon 店铺
      │   ├→ Redis读上次执行时间 → should_run_now判断
      │   └→ 到时间 → run_ai_analysis()
      │       ├→ config_resolver: 解析生效配置
      │       ├→ pricing_engine.run_pricing_analysis()
      │       │   ├→ 获取Ozon实时出价(Performance API)
      │       │   ├→ 三层数据采集(历史+店铺基准+今日)
      │       │   ├→ stage_detector: 商品阶段判断
      │       │   ├→ promo_detector: 大促周期识别
      │       │   ├→ 构建增强Prompt → DeepSeek分析
      │       │   ├→ 安全护栏校验(min/max/调幅/取整)
      │       │   └→ 写入ai_pricing_suggestions
      │       └→ auto_execute?
      │           ├→ Yes → approve_suggestion()
      │           │   ├→ Redis冷却检查(20分钟)
      │           │   ├→ Ozon API修改出价(纳卢布)
      │           │   ├→ 写AdBidLog
      │           │   └→ 企微通知(执行结果)
      │           └→ No → 企微通知(待确认建议)
      │
      └→ 遍历 WB 店铺（低谷期跳过）
          └→ wb_pricing.run_wb_ai_analysis()
              ├→ 同样的三层数据 + DeepSeek分析
              ├→ 写入suggestions(带WB后台直链)
              └→ 企微通知(每条带"去WB改价"链接)
```

## 四、核心算法

### 4.1 五时段策略

| 时段 | 莫斯科时间 | 巡检间隔 | 调价方向 | 调幅范围 |
|------|-----------|----------|----------|----------|
| 上午高峰 | 10:00-14:00 | 30分钟 | 加价 | 5%-20% |
| 晚间高峰 | 19:00-23:00 | 30分钟 | 加价 | 8%-25% |
| 早晨平稳 | 07:00-10:00 | 2小时 | 中性 | 0%-10% |
| 下午平稳 | 14:00-19:00 | 2小时 | 中性 | 0%-10% |
| 深夜低谷 | 23:00-07:00 | 2小时 | 降价 | 30%-50% |

### 4.2 商品生命周期阶段

```
判断优先级：数据不足 > 冷启动 > 衰退 > 测试 > 放量

unknown     数据<3天      → 保守处理，调幅≤10%
cold_start  订单<20单     → 保曝光，不因ROAS降价
testing     CTR≥2% CR<2%  → 降价减消耗，等CR改善
growing     CTR≥2% CR≥2%  → ROAS达标加价，低了适度降
declining   ROAS连续7天↓20% → 收缩预算，控制亏损
```

### 4.3 大促周期（日历驱动）

```
大促前1天(预热)  → 出价×预热系数(1.3)，锁定排名
大促当天(冲刺)   → 出价×冲刺系数(1.7-2.0)，忽略ROAS限制
大促后1天(恢复1) → 出价×0.9，开始降价
大促后2天(恢复2) → 出价×0.95，继续恢复
大促后3天(恢复3) → 出价×1.0，恢复正常
```

### 4.4 三层数据融合

```
第一层：活动历史数据（权重60%）
  - 动态窗口：数据≥14天用21天，≥7天用14天，<7天用全部
  - 排除大促异常数据（妇女节3/6-10、新年12/28-1/5）

第二层：店铺整体基准（权重30%）
  - 今日/7天平均ROAS
  - 用于判断"是单个活动差还是整体差"

第三层：实时数据
  - 今日花费/ROAS/订单/预算消耗率
```

### 4.5 决策优先级（从高到低）

1. 大促周期策略（覆盖一切）
2. 预算控制（消耗>85%且<20:00）
3. 商品阶段策略
4. 时段策略（高峰/平稳/低谷）
5. ROAS目标优化

### 4.6 安全护栏

```python
def validate_suggestion(suggested_bid, current_bid, config, time_strategy):
    1. 最低出价检查 → max(bid, 3)  # Ozon最低3卢布
    2. 最高出价检查 → min(bid, max_bid)
    3. 单次调幅检查 → 限制在±max_adjust_pct%
    4. 取整          → round(bid)  # Ozon要求整数
    5. 再次确保≥3卢布
```

### 4.7 冷却机制

- Redis key: `ai_pricing:cooldown:{campaign_id}`
- 冷却时间: 20分钟（Ozon平台建议）
- Redis TTL: 25分钟
- 冷却中的活动跳过调价，不阻塞其他活动

## 十、WB建议模式
