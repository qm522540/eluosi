## 四、健康诊断翻译"翻译中..."死循环（下午）

### 4.1 用户场景：展开商品行后大量关键词显示"翻译中..."不动（`eee309b`）

诊断三步：
- 先看代码：`HealthProductsTable.jsx` 的翻译逻辑用 `kwTranslations[v]` + `hasZh = zh && zh !== v`
- 服务器实测 5 个词 1.4s 全 OK → 翻译接口本身正常
- 看代码 `.catch(() => {})` 静默吞错 — **三种状态都被同一文案掩盖**：
  1. 真正调中
  2. 失败/超时（catch 吞掉）
  3. Kimi 兜底返原词（zh === v）

**修复**：
- 加 `kwTranslateStatus` state 机：`pending` / `done` / `failed`
- 拆批 30 词一组（避免单次 100+ 词 Kimi 60s 超时整批挂）
- 跳过已翻译词（展开 → 折叠 → 再展开不重复请求）
- catch 加 `console.error` 便于线上排查
- 失败显示红色"未翻译 ↻"可点重试单条
- 手动编辑翻译后自动标 'done'

### 4.2 用户反馈"还是大量未翻译" → 服务器深挖（`6c76384`）

预热脚本跑 Pt.Gril 784 词时日志暴露**两个根因**：

**根因 1（主因）**：Kimi 高频返回 fancy quotes：
```
[
"女性",
"人造花",
"女士手镯 首饰",
...
]
```
注意是中文双引号 `"…"` 而不是 ASCII `"…"`，Python `json.loads` 直接挂 → `_translate_batch` 兜底返原词 → 前端显示"未翻译"。

**根因 2**：Moonshot 429 限流 — 高频翻译触发 RPM 上限。

修复：
- `_call_ai` 加 `_FANCY_QUOTES_TRANS = str.maketrans({"“": '"', "”": '"', "‘": "'", "’": "'", "«": '"', "»": '"', ...})`，json.loads 前 `.translate(...)` 全部转 ASCII
- `BaseAIClient.chat` 加 429 重试：1s → 2s → 4s 指数退避，优先用 `Retry-After` header，最多 3 次
- 其他 HTTP 错误立即抛（避免无限重试）

**实测对比**：
| 跑次 | 结果 |
|---|---|
| 修复前 | OK=264 / FAIL=330（30 批中 JSON 错 + 后期 429 全挂）|
| 修复后 | **OK=782 / FAIL=2（98.5%）** |

部署后跑预热把 Pt.Gril 779 / 784 词全部翻译入库，剩 5 个是 Kimi 真翻不出的生造词。

## 十、今日体会

1. **"用户已踩 N 次 = 必然要做的事"**
   SEO 引擎 daily beat 这事，记忆里 04-24 标了"待议"，4-23 / 4-24 / 4-26 各踩一次，今天用户第 3 次踩到对比题 "WB-E0004 能显示为啥 PT-N0017 不能" 才拍。**用户主动拍前我应该再主动一些** — 同类问题第二次出现就该回头评估自动化。

2. **"catch 不能静默吞错"** — 翻译"翻译中..."死循环根因是 `.catch(() => {})` 把超时 / 401 / 网络错全藏起来，用户等了 30s 还是不知道为什么。**异步任务的 catch 至少要 console.error**，不要图前端清爽就让用户体验出错。

3. **"接口契约 vs 兜底数据 vs 显示文案"三件事别混在一起判**
   `hasZh = zh && zh !== v` 这一行同时包含了"有数据"+"翻译成功"两层语义，所以用户看到的"翻译中..."实际可能是三种完全不同的状态（未发起 / 发起中 / 失败兜底返原词）。**显式 state 机 (pending/done/failed) > 隐式判断**，否则你永远不知道用户看到的是哪种。

4. **"Kimi 返中文引号"是个值得记入团队规范的坑**
   Moonshot 默认 temperature=0.2 也照样高频返 fancy quotes（不是温度问题，是模型 finetune 习惯）。这个坑之前老林写 `_call_ai` 时没考虑，今天碰到才修。**所有调 AI 返 JSON 的地方都该加 fancy quotes 兼容**，建议老林后续走 review 时统一抽。

5. **"DB 实测 vs 文档推理"**
   今天 PT-N0017 vs WB-E0004 的对比，光看代码 + memory 推理"是不是引擎没跑"是不够的，必须 `psql` 进去查 `seo_keyword_candidates.product_id, sources` 的真实分布才能判出"时间错配"这个根因。**复杂跨表问题先 SQL 实测，别陷入代码分析**。
