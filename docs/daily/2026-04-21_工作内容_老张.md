## 四、方向 A：关键词表现追踪 全栈落地（4 commit ~750 行）

### A-Step 1 — 后端 service + 2 个 API `0166f62`

`app/services/seo/keyword_tracking_service.py`（~330 行）：

**核心函数 `compute_keyword_tracking`**：
- 基于 `today` 锚定日期段：本期最近 N 天 vs 上一个 N 天（默认 7）
- 单条聚合 SQL：SUM(CASE WHEN stat_date BETWEEN ...) 一次性算出 imp/clk/cart/ord/rev 的本期和上期
- Python 层算趋势 + 预警

**趋势分类 6 档**：`new / up / stable / down / vanish / idle`
- prev=0 cur>0 → new（新词）
- cur=0 prev>0 → vanish（消失）
- |delta| < 20% → stable；≥ +20% → up；≤ -20% → down
- prev=cur=0 → idle

**预警规则 3 档**：
- **vanish**：上期曝光 ≥ 50 且本期 0 —— 紧急
- **drop**：上期曝光 ≥ 20 且本期/上期 ≤ 0.7 —— 掉 30%
- **orders_drop**：上期订单 ≥ 2 且本期 0 —— 转化断崖

**数据就绪度检查**：预查询 `COUNT(*) WHERE stat_date BETWEEN prev_start AND cur_end`，= 0 时返 `data_status='not_ready'` + 平台专属引导：
- WB → "POST /api/v2/search-report/product/search-texts，需 Jam 订阅"
- Ozon → "POST /v1/analytics/product-queries/details，需 Premium 订阅"
- Yandex → "Yandex Market 暂不支持商品级搜索词洞察"

**单词下钻 `list_query_top_skus`**：
- 给一个 query_text 返回 Top N 带该词曝光/订单的商品
- 用于前端下钻 Drawer："这词靠哪几个商品撑起来"

**新接口**：
| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/seo/shop/{id}/keyword-tracking` | 主列表 + 环比预警 |
| GET | `/seo/shop/{id}/keyword-tracking/skus` | 单词下钻 Top N 商品 |

**合规自查**：
- 规则 1 tenant_id：4 处 SQL WHERE 全带 ✓
- 规则 2 datetime：用 `datetime.now(timezone.utc).date()` ✓
- 规则 4 shop_id：WHERE 全带，API 层 `Depends(get_owned_shop)` 守卫 ✓

## 五、方向 B：Before/After ROI 报表 全栈落地（2 commit ~600 行）

### B-Step 1 — 后端 service + API `234e871`

`app/services/seo/roi_report_service.py`（~235 行）：

**核心逻辑**：
1. 拉所有 `approval_status='applied'` 记录，按 `applied_at DESC`
2. 对每条：切割点 = `applied_at.date()`
   - Before 窗口：`applied_at - window_days ~ applied_at - 1`（闭区间，不含切割点当天）
   - After  窗口：`applied_at ~ applied_at + window_days - 1`（clamp 到今天）
3. 从 `product_search_queries` 按 `platform_sku_id` 聚合 imp/ord/rev
4. 状态分类：
   - `completed`：applied_at 距今 ≥ window_days
   - `observing`：applied_at 距今 < window_days（后窗还没满）

**delta_pct 设计**：before=0 → null（前端显示"新基线"），避免除零 + 显式区分"真无数据"和"零增长"

**新接口**：`GET /seo/shop/{id}/roi-report?window_days=14`（3-60）

**空态引导**：0 条 applied 时返 `empty_hint` 文案指引"去 AI 生成历史 Tab 点标记已用"

**合规自查**：
- 规则 1 tenant_id：applied_sql / stats_sql 两处全带 + JOIN tenant 匹配
- 规则 4 shop_id：applied_sql `pl.shop_id = :sid` + stats_sql `shop_id = :sid`
- 规则 2 datetime：`datetime.now(timezone.utc)` + 兼容 DB 返 naive DATETIME 补 tzinfo

## 十一、今日体会

1. **"Probe 先行"再次兑现价值**
   今天如果不先查 `product_search_queries` 的 median_position 全 null 就直接按"排名追踪"的字面意思开工，列名/state/API 全都会围绕 position 构建。Probe 3 分钟省掉至少 2 小时的"发现后重构"。**对接新接口 / 做新功能前先 probe 真实数据** —— 这条规则已经稳稳刻进肌肉记忆。

2. **"两个全壳功能 + 迷茫用户" = 先做出来让用户选**
   用户说"我现在也有点迷茫"。这时候最差的做法是继续讨论细节、画原型；最好的做法是**做出能跑的东西让用户看**，因为用户能指着实物说"这个不对" 比指着概念说"我要什么"容易得多。今天 5 commit ~1350 行落地两个功能，用户明天打开就能摸到按钮、点 Tab、看数据。**迷茫时产出 > 规划**。

3. **"菜单终于名副其实" 比"加一个子菜单"更重要**
   Report 菜单叫"效果报表"但内容只是"AI 生成历史"，这是产品设计债。今天加 Tabs 让 Tab 1 成为真正的"效果报表"（Before/After ROI），Tab 2 作为历史回溯附属功能 —— **名实相符比功能多**。如果继续加一个新菜单项"ROI 对比"，菜单 5 项拥挤，反而不如用 Tabs 压到已有菜单。**产品的克制比产品的扩张难**。

4. **"空态设计 = 引导下一步"**
   ROI 报表的 `empty_hint` 字段不是应付 Empty 组件，而是**后端告诉前端"用户下一步该去哪"**。applied=0 → hint 文案直接说"去 AI 生成历史 Tab 点标记已用"。用户不需要猜为什么页面是空的，也不需要翻文档。**好的空态比非空态体验更值得精心设计** —— 因为用户第一次进来一定是空的。

5. **"状态分类枚举值的命名"决定 UI 清晰度**
   Tracking 的趋势 6 档 `new/up/stable/down/vanish/idle` + 预警 3 档 `drop/vanish/orders_drop` —— 每个词都是简单英文单词，前端 switch case 写起来不需要查表。如果我写成 `status1/status2/status3` 或中文拼音 `xinci/shangsheng` 前端就得维护一个 i18n 映射。**枚举值命名的简洁直白比中文更贴合代码**，中文留给 UI label。
