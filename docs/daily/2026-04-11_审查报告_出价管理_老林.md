## 二、严重问题 / BLOCKER（11项）

### 2.1 多租户隔离漏洞群（覆盖 11+ 个接口/函数）

**漏洞本质**：所有表的 WHERE 条件只用 `shop_id`，没加 `AND tenant_id = :tenant_id`。攻击者只要枚举 shop_id 就能读/写其他租户数据。

#### 后端 API 路由层（`app/api/v1/bid_management.py`）

| # | 接口 | 漏洞位置 | 危害 |
|---|------|---------|------|
| **#1** | `GET /time-pricing/{shop_id}` | 行 89-91 `WHERE shop_id = :shop_id` | 泄露其他租户分时调价规则 |
| **#2** | `GET /ai-pricing/{shop_id}` | 行 203-206 `WHERE shop_id = :shop_id LIMIT 1` | 泄露 AI 模板参数、启用状态 |
| **#3** | `GET /suggestions/{shop_id}` | 行 306-307 `where = ["s.shop_id = :shop_id", ...]` | 泄露 AI 建议 含 ROAS/出价/SKU名 |
| **#4** | `GET /bid-logs/{shop_id}` | 行 460 `where = ["shop_id = :shop_id"]` | 泄露全部出价调整历史 |
| **#5** | `GET /data-status/{shop_id}` | 行 534 `WHERE shop_id = :shop_id` | 泄露数据采集状态 |

#### Service 层（`app/services/bid/*`）

| # | 函数 | 位置 | 危害 |
|---|------|------|------|
| **#7** | `time_pricing_executor.update_rule` | 行 254-285 `INSERT ... ON DUPLICATE KEY UPDATE`，UNIQUE KEY 是 `shop_id` | A 租户传 B 家 shop_id，`ON DUPLICATE` 触发后覆盖 B 的规则（而且不更新 `tenant_id`，数据污染） |
| **#8** | `time_pricing_executor.enable` | 行 290-314 签名有 `tenant_id` 参数但**完全没用**，SQL 只 `WHERE shop_id = :shop_id FOR UPDATE` | 任意用户启用别家分时调价 |

**昨日日报第1轮审查已经发现并修复过同类漏洞**（`approve_suggestion 多租户隔离漏洞`），老张这次又犯。这是**严重的规范执行问题**。

#### 统一修复方案（推荐）

新增 `app/dependencies.py` 的一个 Depends，在路由层一次性校验：

```python
from fastapi import Depends, HTTPException
from app.models.shop import Shop

def get_owned_shop(
    shop_id: int,
    db: Session = Depends(get_db),
    tenant_id: int = Depends(get_tenant_id),
) -> Shop:
    shop = db.query(Shop).filter_by(id=shop_id, tenant_id=tenant_id).first()
    if not shop:
        raise HTTPException(404, detail={"code": ErrorCode.SHOP_NOT_FOUND, "msg": "店铺不存在"})
    return shop
```

所有带 `{shop_id}` 路径参数的路由改为：

```python
@router.get("/time-pricing/{shop_id}")
def get_time_pricing(
    shop: Shop = Depends(get_owned_shop),   # 进来就是已校验的
    db: Session = Depends(get_db),
):
    # 后续用 shop.id / shop.tenant_id 即可
```

**好处**：一次写，所有接口受益。service 层仍需补 `tenant_id` 参数做双重保险（纵深防御）。

**service 层的 `approve_suggestion / reject_suggestion / approve_batch / reject_batch` 必须改签名**：

```python
async def approve_suggestion(db, tenant_id: int, suggestion_id: int) -> dict:
    row = db.execute(text("""
        SELECT ... FROM ai_pricing_suggestions s ...
        WHERE s.id = :id AND s.tenant_id = :tenant_id
    """), {"id": suggestion_id, "tenant_id": tenant_id}).fetchone()
```

**11 个隔离漏洞必须全部打掉**，不能遗漏任何一个。二次审查会随机挑两个接口验证。

### #12 `datetime.utcnow()` 已弃用 + naive datetime 时区混乱

**位置**：
- `app/models/ai_pricing.py:44,46,81,101,103,134` — 模型默认值
- `app/services/bid/ai_pricing_executor.py:169` — `start = datetime.now()`
- `app/services/bid/ai_pricing_executor.py:440` — `now = datetime.now()` 写入 `executed_at`

**严重性**：Python 3.12+ 抛 DeprecationWarning；与 Celery `enable_utc=True` 冲突；`generated_at`（模型默认 utcnow）和 `executed_at`（naive now）时区不齐，"次日过期"判断会错。

**昨日日报第2轮审查明确修过同一问题**，这次又回退。

**修复**：

```python
from datetime import datetime, timezone

def _utc_now() -> datetime:
    return datetime.now(timezone.utc)

# 模型
created_at: Mapped[datetime] = mapped_column(
    DateTime, nullable=False, default=_utc_now,
)
```

### #14 `moscow_time._iso()` 时区假设错误

**位置**：`app/utils/moscow_time.py:183-199`

```python
# 数据库里 last_executed_at 是 naive datetime（CURRENT_TIMESTAMP）。
# 服务器假设按 UTC 存。这里加上 UTC tzinfo 再转莫斯科。
```

**但实际代码做的是**：

```python
if dt.tzinfo is None:
    dt = MOSCOW_TZ.localize(dt)   # ← 按莫斯科时区处理
```

**注释和代码矛盾**。MySQL `NOW()` 返回的是 MySQL 服务器时区（默认 UTC 或系统时区）。如果生产机 MySQL 是 UTC：
- 注释正确（应该按 UTC localize 再 convert）
- 代码错误（按莫斯科 localize 会让显示时间多 3 小时）

老张必须确认生产 MySQL 的 `time_zone` 配置，并让注释和代码一致。推荐**统一按 UTC 存并显示时转莫斯科**：

```python
if dt.tzinfo is None:
    dt = pytz.UTC.localize(dt)
dt = dt.astimezone(MOSCOW_TZ)
return dt.isoformat()
```

### #17 `restore_sku` 匹配 SKU 使用 LIMIT 1 可能漏恢复

**位置**：`time_pricing_executor.py:331-342`

```sql
WHERE c.shop_id = :shop_id AND ag.platform_group_id = :sku
LIMIT 1
```

同一个 SKU 可能在多个活动下都有 `ad_group` 记录。`LIMIT 1` 只恢复一个，其他活动里的 SKU 出价依然是分时调价过的值，和用户预期不符。

**修复选项**：
- 去掉 LIMIT 1，遍历所有匹配的 ad_group 都恢复
- 或在 API 加 `campaign_id` 参数精确定位

### #18 AI `analyze_now` 无并发保护

**位置**：`ai_pricing_executor.py:164`

手动触发 `POST /ai-pricing/{shop_id}/analyze` 时没有任何锁。用户连点两次或多 Tab 同时触发会：
- 并发调 DeepSeek API（费钱）
- 并发写入 `ai_pricing_suggestions`（数据重复）

**修复**：用 Redis `SETNX` 限制同店铺 60 秒内只能触发一次：

```python
redis_key = f"bid:analyze_lock:{shop_id}"
if not r.set(redis_key, "1", nx=True, ex=60):
    return {"code": ErrorCode.BID_DATA_SYNC_RUNNING, "msg": "分析进行中，请等待 60 秒"}
```

### #19 `execute()` 的 `retry_at` 时区比较 naive

**位置**：`ai_pricing_executor.py:158`

```python
if cfg.retry_at and now_moscow().replace(tzinfo=None) < cfg.retry_at:
```

`now_moscow()` 带莫斯科时区 → `.replace(tzinfo=None)` 剥掉得到莫斯科小时 naive；而 `cfg.retry_at` 是写入时的 `datetime.now()`（服务器本地时区 naive）。如果服务器时区 ≠ 莫斯科（通常 = UTC），判断会差 3 小时。

**修复**：统一用 UTC 存 + UTC 比较：

```python
if cfg.retry_at and datetime.now(timezone.utc).replace(tzinfo=None) < cfg.retry_at:
    ...
```

配合 #12 的修复。
