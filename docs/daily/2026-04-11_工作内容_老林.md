### 3. 时区显示 bug 修复（晚上 — 本 session 主要工作）

**用户报告**：刚刷新的店铺，"X 小时前同步"显示"9 小时前"。

**根因诊断**：
- 写入端：`app/tasks/ad_tasks.py` ×3、`app/services/data/ozon_stats_collector.py` ×5 用 `datetime.utcnow()` / `datetime.now()` 写入 naive datetime
- 序列化端：`app/services/shop/service.py` 和 `app/api/v1/bid_management.py` 直接 `.isoformat()` 输出无时区后缀的 ISO 串（如 `2026-04-11T05:00:00`）
- 前端 `new Date(str)` 对无 tz 后缀的串按本地时区 (China UTC+8) 解析 → 时间往前移 8 小时 → 显示"8/9 小时前"
- 这是 CLAUDE.md 规则 2 的典型违反

**修复**（commit `1d2ed69`）：
| 位置 | 修改 |
|---|---|
| `app/services/shop/service.py:189` | 新增 `_iso_utc()` 辅助，naive 自动按 UTC 附加 `+00:00` |
| `app/api/v1/bid_management.py` data-status 接口 | 同上模式（局部 helper） |
| `app/tasks/ad_tasks.py` ×3 | `datetime.utcnow()` → `datetime.now(timezone.utc)` |
| `app/services/data/ozon_stats_collector.py` ×5 | `datetime.now()` → `datetime.now(timezone.utc)` |
| `app/api/v1/ads.py:986` | MySQL `NOW()` → 绑定 Python tz-aware datetime 参数（消除对 MySQL 服务器 time_zone 的依赖） |

**部署**：`bash deploy_remote.sh --backend` 完成，三进程重启 OK。
