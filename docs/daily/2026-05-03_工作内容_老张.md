### 4.4 老板问"定时任务拿几天数据"
**默认 7 天**, 三处定值:
- `wb_search_texts_task.py:67` `refresh_shop(..., days=7)`
- `wb_search_texts_task.py:146` `def sync_wb_search_texts_for_shop(..., days: int = 7)`
- `search_insights/service.py:308` `def refresh_shop(..., days: int = 7, ...)`

窗口算法: `today-8 ~ today-2` (留 1 天 buffer 给 WB Jam 数据延迟). **按缺失天补**: DB 已有的天不重拉, 调 0 个 API; 缺哪天调那天 1 个 24h 窗口请求.

### 4.5 老板问"WB 自动屏蔽托管"是啥
活动级关键词自动 minus 托管: 每日 MSK 04:30 跑 `auto_exclude_keywords`, 用租户 efficiency_rules 找 waste 词加进活动 minus, 跳过白名单+已屏蔽, 写日志含节省金额估算. 入口在广告管理-推广信息-活动详情抽屉里. **当前没人开启任何活动**, 所以每天 04:30 跑等于空转. 暂保现状不动.

### 4.6 老板痛点反馈"出价管理 PT.Gril 数据源 45→36 天, 新的没拿到老的反被删"
**根因**: `wb_stats_collector.smart_sync` 末尾**无条件**调 `_clean_old_data` (line 105), 不管同步成功失败. PT.Gril 因 WB seller quota 持续 429 burnout, 4-24~5-2 新数据一直拉不到, 但每次 smart_sync 末尾仍 clean → 净亏 9 天 (45 - 9 = 36).

**修法 (老板 A 方案)**: 改条件触发 `if total_synced > 0` 才 clean, Ozon 同思路挪到后台线程末尾按 `total_updated > 0`. **下次 quota 仍烧光时同步失败 → skip clean → 36 天保持不动等恢复**. commit `ae75918`.

### 4.7 老板问"数据源更新是拉 1 个月内空的数据吗"
**不是**. ad_stats 是 **45 天滚动窗口 + 按缺失天补**. PT.Gril 现状缺 4-24~5-2 共 9 天, 每次点同步只对这 9 天调 API 而非无脑拉 45 天. 跟搜索词洞察 (90 天 / 7 天默认) 是同一思路 (4-26 老林两边一起重构的).
