## 三、4 个 bug 修复 + 部署

### 3.1 fix(search-insights): WB 早退保护单天 reset + day_total>0 豁免（commit `89bd355`）

文件：`app/services/search_insights/service.py`

**原 bug**：`EMPTY_BATCH_ABORT` 跨天累计 + 不看 `day_total` → shop=6 PT.Gril 4-20 第 1 批写 9 行后第 2-4 批返空（**真实稀疏**），整体 break 退出，后 9 天完全没尝试。

**改动**：
- `consecutive_empty` 单天 reset（避免跨天累计误判稀疏 shop）
- 触发条件加 `and day_total == 0`（当天 API 已工作就不早退）
- 改为 `break` 当天 inner loop，外层继续下一天
- 新增 `EMPTY_DAYS_ABORT=3` 跨天兜底（连续 3 整天 0 行才整体退出）

**实测验证**（shop=6 重跑 days=10）：
- 4-20 保留之前 9 行
- 4-21/4-22/4-23 连续 3 天前 3 批空 + 0 行 → 跳过当天 + 累计 3 天 0 行 → `EMPTY_DAYS_ABORT` 触发整体退出
- **业务结论**：PT.Gril 真稀疏，4-21~4-23 这 3 天平台真没用户搜词命中（不是 bug）

### 3.3 fix(seo): keyword-rollup 商品数门槛缺失（commit `79e189f`）

文件：`app/services/seo/keyword_rollup_service.py`

**原 bug**：`pc_sql` 仅 `COUNT(DISTINCT product_id)` 无门槛 → "28 天有过 1 次零散曝光"也计入。
实测："серьги из медицинского сплава" + ANI-OZON+Pt.Gril 跨店：

| 门槛 | 商品数 |
|---|---|
| 无（旧） | 124 |
| `SUM(impressions) >= 5` | 81 |
| `SUM(impressions) >= 10` | 56 |
| `SUM(orders) >= 1` | **6**（真带单）|

**修法**：嵌套 `GROUP BY (query, product_id)` + `HAVING (SUM(impressions)>=5 OR SUM(orders)>=1)`，跟主表"真贡献"语义对齐。124 → ~80（ANI-OZON 单店 7）。

### 3.4 fix(seo): 关键词翻译 batch API 不再被 Kimi 阻塞（commit `f2401e4`）

文件：`app/services/keyword_stats/translator.py` + `app/api/v1/keyword_stats.py` + `frontend/src/api/keyword_stats.js`

**问题**：前端 KeywordRollupTab 一次发 200 词翻译，其中 47 个没在 DB → 走 Kimi L3 → 60s timeout 内 Kimi 跑不完 → 整批响应 timeout → axios reject → **153 个 DB 命中的也丢了** → 用户看到全部"翻译中..."。

**用户诊断关键**：对照"关键词统计"页面工作正常发现差异——KeywordStats 每页 20 词全 DB 命中，KeywordRollupTab `limit=200` 把问题暴露。

**修法**：
- `translator.py` 加 `db_only` 参数，True 时跳过 L3 Kimi（没命中留原词）
- API `/translate-keywords` 加 `db_only` 字段
- 前端 `translateKeywords` 默认 `dbOnly=true`，timeout 60s → 10s
- 用户面调用永远秒级 DB 响应；Kimi 翻译由独立后台脚本预填

4. **"04-28 P0 又拖了一天"** — 04-28 trip 事故 hook 终极方案承诺 04-29 做，被 SEO 关键词分析连续拉走 3 天。今天又被 SEO 优化建议拖了。明天再不做就累计 4 天。**承诺 P0 必须排在用户当天新需求之前，否则永远做不上**。
