## 七、架构决策记录

### 7.1 为什么 C 先 B 后（推翻原计划）

**原计划依据**：B（SEO 健康分）给"全店鸟瞰 + Top 20 待优化"视图，能解决"看不懂优先级"问题。

**推翻理由**：
- 用户说"看不懂"更可能是"不知道下一步做什么"——B 给 Top 20 还是没告诉用户"然后呢"
- C 的闭环性更强：**勾词 → AI 出标题 → 复制去改 → 14 天看效果**，每一步用户都能做
- C 做完后，B 的 Top 20 每行都能挂「AI 生成标题」按钮，B 就顺势接上（不会浪费工作）

### 7.2 为什么返回字典而非抛异常

对齐 `app/services/seo/service.py` 现有惯例：
```python
return {"code": ErrorCode.SEO_CANDIDATE_NOT_FOUND, "msg": "候选词不存在"}
```

本 service 共 5 个异常分支全改成 return dict，API 层统一 `result.get('code') != 0` 判断，前端 `res.code === 0` 判断。**一致性 > 个人偏好**。

### 7.3 为什么选 max_candidates=30

- GLM-4-Flash 上下文 128k 完全够
- 但电商标题融合 > 10 个词就重复/冗余；> 30 个词 AI 会随机丢词
- 用户体验角度，勾 30 个已经很多，强迫少而精
- 保留余量：用户勾错可 max_candidates=30 兜底（前端又限了 1~30）

### 7.4 为什么不直接写回商品（三期再做）

- **风险控制**：AI 生成标题直接覆盖原标题 → 错了没法回滚（Ozon 商品更新 API 幂等但原值只能从 platform_listings.title_ru 快照恢复）
- **审批流已预留**：`seo_generated_contents.approval_status = 'pending' → 'approved' → 'applied'`，三期只需实现 applied 时调平台 API 改价 + 记录旧 title 到 `original_text` 用于回滚
- **用户信任建立**：一期让用户复制手动改，14 天观察效果；二三期才上"一键写回"
- **实现难度**：WB 和 Ozon 的商品更新 API 字段格式差异大（WB 多语言 + Ozon 属性强约束），统一接口需额外 2-3 天，一期先避开

## 十一、今日体会

1. **"用户的'看不懂'是流程不闭环，不是信息密度低"**
   最初我以为用户"看不懂"是因为数据太多（2909 条）、术语太专业（ROAS/score），想去加 Top 优先级聚合 + 术语 tooltip。实际听完用户说"简单到看不懂"才意识到：**问题不是"看不懂信息"而是"看不懂该干什么"**——候选词表没有端点动作。**这是产品思维和工程思维的分水岭**：工程师倾向信息增补，产品思维倾向动作闭环。

2. **"推翻原计划"不是坏事，是响应用户的信号**
   昨天我列的明日优先级原本 B→C，但用户一句"看不懂"我就把 C 提前。**不要为昨天写的日志辩护**——日志是计划不是承诺，用户一句话就能推翻。老林 04-19 §三十八.1 写"不要为旧 commit 辩护"同理。

3. **AI 基础设施的价值在"不需要重造"**
   我原本以为 AI 接入要从 httpx 封装、错误处理、token 统计、日志持久化全手写，结果老林早就铺好整套 `services/ai/` + `AiDecisionLog` + router。**今天 Step 1 写了 290 行纯业务代码，0 行基础设施代码**。这就是"先建好基础资产再做上层应用"的复利——老林 04-19 §九.5 的体会今天在我身上兑现。

4. **"自己建表 vs 复用老表"的节奏判断**
   本来想建 `seo_ai_generated_titles` 专用表（product_id / candidate_ids / new_title / ...），看了老林的 `SeoGeneratedContent` 发现字段全覆盖 + 还预留了 approval_status 审批流。**复用让工期从 6 小时压到 4 小时**。判断原则：**新建表前先 `ls app/models/` 全扫一遍**，70% 的通用场景老林早建了。

## 三十四、追加后体会

1. **"标记为遗留"≠"真的没做"—— 日志错列的成本**
   §二十八 把早已修完的 3 个 P0 列为"待修 P2"，今天准备动手才发现是错报。**日志是给未来自己看的备忘录，错一行下次就多浪费 30 分钟**。以后列遗留前先 `git log --oneline --grep='关键词'` 过一遍，确认 commit 没做过。

2. **写用户文档 ≈ 最后一次产品走查**
   本以为写操作指南是"把功能翻译成中文"，真写的时候发现 "Step 3 勾选跨商品的词系统提示什么" 这种细节我自己都得回看代码。**写文档强迫你把每一步都亲自过一遍**，等于又走查了一次产品。这个过程发现 `/seo/report` "标记已用" 的定位文字可以更明确（一期先这样，二期 ROI 对比时再改）。

3. **测试补的时机——等功能稳定后立刻补，不要等"有空再写"**
   SEO 一期今天刚稳，明天 2 周内用户实测会出 bug。现在 29 个单元测试就是 **回归护栏** —— 明天修 bug 时至少知道"动 A 接口没影响 B 接口"。**"有时间再写测试"永远不会发生**，稳定期就是最好的时机。

4. **远程跑测试是正解**
   本地 Windows Python 没 FastAPI，折腾 venv 至少 10 分钟。直接 `ssh + pip install pytest + pytest` 在远程跑，5 分钟搞定包括 commit+push+pull+run+看结果。**生产环境就是最好的集成测试环境**（当然前提是测试不写数据库 —— 我全 mock 了 service 层所以安全）。

5. **"5 接口测试" vs "service 层测试"**
   一期只补了 API 层（29 用例），service 层的算法（evaluate_paid_to_organic / compute_shop_health / _finalize_score）还 0 覆盖。**API 层测 "接口合约"，service 层测 "算法正确性"**。今天只做前者是因为后者依赖 DB，需要起 test DB 或更重的 mock。留给 B 项。
