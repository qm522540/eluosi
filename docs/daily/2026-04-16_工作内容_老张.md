### 4. 一键初始化（核心产品逻辑）

用户反馈："先抓取 WB 平台的分类、属性，然后把本地分类属性都跟 WB 一样，也同时抓取 OZON 的分类和属性，然后本地和 WB/OZON 进行关联"。

#### 4.1 批量翻译工具 `_translate_batch`（`6383e44`）

俄文→中文，一次 200 条，去重后翻译再按原顺序填回。AI 异常时用原文兜底。

#### 4.2 `init_mapping_from_wb`（`6383e44`）

流程：
1. 从 `platform_listings` 反查店铺已用 `subjectID` 集合
2. 拉 WB `object/all` 全量分类字典筛选
3. 批量翻译俄文→中文
4. 建 `local_category` + WB 品类映射 `is_confirmed=1`（1:1 直接对齐）
5. 对每个分类拉 `charcs` 属性列表，翻译后建 WB 属性映射
6. 枚举值同步翻译写入

**实测**（shop_id=1 WB-Shario）：3 本地分类（气球/雨滴/节日套装）+ 62 WB 属性映射。

#### 4.3 `init_mapping_from_ozon`（`4abe526`）

与 WB 不同——Ozon 是"智能扩充"：
1. 反查 OZON 店铺已用 `description_category_id`
2. 从商品 `product_info` 反查 `type_id`（存 platform_listings 没存 type_id，即时反查）
3. 从 `category_tree` 反查名称+面包屑
4. **AI 归一去重**：一次调用判断每个 OZON 分类对应哪个已有本地分类（或 null=新建）
5. 有对应 → 只建 OZON 映射（`is_confirmed=0` 待确认）
6. 无对应 → 新建本地分类（翻译中文）+ OZON 映射（`is_confirmed=1` 1:1）
7. 属性"同名复用"：本地已有同名属性（WB 种子带来）直接复用 `local_attr_name`，只新增 Ozon 映射行

**实测**（shop_id=2 OZON-Shario）：
- 本地分类 3→4（AI 合并 1 个，新建 1 个"时尚饰品 Бижутерные украшения"）

- Ozon 属性映射 80 条（1 条同名复用，证明智能合并生效）

#### 4.4 `match_ozon_from_local`（`6383e44`）

## 六、今日体会

1. **潜伏 bug 的触发点往往是"较真的用户"**：今天两个大 bug（`listings[]` 缺失、WB 价格全是 0）其实 04-15 就存在，前端页面显示"价格 ₽0"或展开行是空的，但没人觉得不对。直到用户今天问"销售价为啥没显示""价格是什么价格"才扒出来。写代码的假设一定要拿真数据过一遍。

2. **AI 推荐必须是"推荐"不是"决定"**：今天映射管理用了大量 AI（翻译、归一、匹配），但每条 AI 结果都有 `ai_confidence` + `is_confirmed` 两道闸门，用户前端看到的是置信度徽章+待确认状态。宁愿让用户多点一下确认，也不让 AI 静默污染生产数据。

3. **一次 AI 调用比多次强**：`_ai_merge_ozon_to_local` 一次把所有 OZON + 所有本地分类给 AI 返回 idx 数组，比循环每个分类都调一次快 N 倍、省 token。前提是 AI 能稳定按顺序返回等长数组（temperature=0.2 + 明确长度要求）。

4. **"AI 返回 idx 而非 id"是稳定性技巧**：早期版本让 AI 直接返回 `{"id": "123456", ...}`，AI 经常把 id 拆字段、少位、加前缀。改成让 AI 返回序号 idx，后端按 idx 查原数组拿 id，再也没出过字段错位问题。

5. **双 ID 表设计看似冗余实则必要**：Ozon 分类用 `description_category_id + type_id` 两个 ID，早期我觉得加个 `platform_category_extra_id` 字段"丑"，想着用 JSON 或拼字符串。实测后发现分字段最省事——AI 不用解析字符串、前端不用拆分、ORM 天然支持。多一个字段远比多一处解析逻辑便宜。

6. **`init-from-wb + init-from-ozon` 是"种子 + 扩充"范式**：以 WB 作为本地分类的骨架（1:1 翻译），OZON 来的时候先 AI 判断能不能并入已有骨架。这个范式比"每个平台独立建本地分类再让 AI 找对应"高效多了——因为 WB 侧天然对齐，不用确认。
