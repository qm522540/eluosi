### 4.2 后端 auto_exclude 加 per-SKU 亲和度门控（`ad_auto_exclude_task.py`）

**之前**：177 候选词 × campaign 级 waste 判定 → 对所有 SKU 一刀切地加屏蔽
**现在**：加两道门控：

```python
# 门控 1：SKU 活动内曝光占比
if per_sku[nm_id].share < 0.03:  # 占比 < 3%
    continue  # 低流量 SKU 不值得瞎堆屏蔽词

# 门控 2：关键词与 SKU 标题亲和度
affinity = _kw_sku_affinity(keyword, sku_title)  # 0~1
if affinity < 0.5:  # 至少一半 token 能前缀命中 SKU 标题
    continue  # WB 拍卖匹配是标题驱动，不相关的词根本不会触发该 SKU
```

**亲和度算法**：关键词 token（去短词）取前 4 字作俄语词根，检查在 SKU 标题 lowercase 里能否 substring 命中。例：

- SKU "Серьги-сердечки из медицинского сплава"
- 关键词 "серьги для детей" → `серь` ✓ / `для` ✓ / `дет` ✓（标题里有 "медицин") × → 2/3 = 67%，超阈值，允许屏蔽

**日志增强**：`AdAutoExcludeLog.reason` 追加 `亲和度 67% · SKU占比 45%`，方便排查。

**参数可配**：
- `per_sku_min_affinity`（默认 0.5）
- `per_sku_min_share`（默认 0.03）
放在租户 `efficiency_rules` 里，后续可按租户调阈值。

### 33.2 核心 Bug 根因（类目扩散数字继承）

`app/services/seo/service.py` 的"类目推荐"逻辑：

- **后端 SQL**：聚合时 `JSON_CONTAINS(sources, JSON_OBJECT('scope','self'))` 条件 SUM，category-only 行不计入

## 三十六、本轮体会（4 条）

1. **"用户反馈驱动的打磨密度比我自己 review 高 10 倍"**
   从"自然·本商品什么意思"到"店铺多选"—— 7 轮反馈挖出 2 个真 bug（类目继承 + SUM 放大）+ 3 个文案问题 + 2 个交互缺失。这些我自己 code review 永远想不到。**用户的困惑点 = 产品的盲点**。今日连环反馈算是一整套交互可用性测试。

2. **"数字错觉 = 产品事故"**
   76 单（真 2 单）、3952 曝光（真 104 曝光）—— 这不是小数点位数问题，是**用户决策被误导**。继承的数字在单行展示时没事（同一份 2 单被复制展示给每个商品都是"这词对它有用"的信号），**SUM 瞬间变灾难**。任何聚合场景都要检查 SUM 的底层行是不是"独立真实"的。规则：**SUM 前先问"这些行的数字独立吗"**。

3. **"文案是产品的最小可运行语义"**
   "暂无实证·待试水" / "类目推断·待试水" / "0 曝光·系统推荐加词" —— 同一个状态我今天改了 3 次文案。第 3 次用户说"现在懂了"。关键是从"抽象概念"（实证 / 推断）改到"具体事实"（0 曝光 / 系统推荐加词）—— **用户读产品就是在读文案，抽象一分钟产品就不能用**。后续写 Tag/Tooltip 直接从"具体数字 + 具体动作"入手。

4. **"双模式切换 = 用保留原能力 × 上新能力"**
   按商品看的 AI 标题生成闭环我不敢动（原产品核心价值），但新关键词聚合视图又是用户要的新形态。最后用 `productFilter` state 作为开关：有 → 扁平保 AI 闭环，无 → 聚合看全景。**新旧形态并存，用一个 state 无缝切换**，比推倒重来更稳。这个双模式模式以后遇到类似需求能复用。
