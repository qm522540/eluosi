### 4.2 后端 auto_exclude 加 per-SKU 亲和度门控（`ad_auto_exclude_task.py`）

**之前**：177 候选词 × campaign 级 waste 判定 → 对所有 SKU 一刀切地加屏蔽
**现在**：加两道门控：

```python
# 门控 1：SKU 活动内曝光占比
if per_sku[nm_id].share < 0.03:  # 占比 < 3%
    continue  # 低流量 SKU 不值得瞎堆屏蔽词

# 门控 2：关键词与 SKU 标题亲和度
affinity = _kw_sku_affinity(keyword, sku_title)  # 0~1
if affinity < 0.5:  # 至少一半 token 能前缀命中 SKU 标题
    continue  # WB 拍卖匹配是标题驱动，不相关的词根本不会触发该 SKU
```

**亲和度算法**：关键词 token（去短词）取前 4 字作俄语词根，检查在 SKU 标题 lowercase 里能否 substring 命中。例：

- SKU "Серьги-сердечки из медицинского сплава"
- 关键词 "серьги для детей" → `серь` ✓ / `для` ✓ / `дет` ✓（标题里有 "медицин") × → 2/3 = 67%，超阈值，允许屏蔽

**日志增强**：`AdAutoExcludeLog.reason` 追加 `亲和度 67% · SKU占比 45%`，方便排查。

**参数可配**：
- `per_sku_min_affinity`（默认 0.5）
- `per_sku_min_share`（默认 0.03）
放在租户 `efficiency_rules` 里，后续可按租户调阈值。

### 9.7 下半场体会

1. **"用户的每一次否定都对应一条隐藏假设"** —— "出现更多了" / "每簇只 4 条" / "为啥全部拒绝" / "核心簇是空的" —— 每个反馈都让我改一层算法。**架构的真实边界是用户实测打出来的，不是设计出来的。**

2. **"oracle 模式是解决'对方有规则不文档化'的万能钥匙"** —— WB 不告诉我们哪些词可屏，我们就用"set-minus 一次测 N 个 + 立即回滚"把规则反推出来。**所有无文档 API 都该有这种探针式反向工程能力。**

3. **"AI 聚类 + 确定性后处理"比纯 AI 可靠** —— DeepSeek 的 prompt 再强也有漂移；token 后处理是客户端规则，100% 确定性。**混合算法（AI 粗筛 + 规则校准）比纯 AI 或纯规则都好。**

4. **"用户给系统补信号"是可行的** —— 手动"+ 添加 WB 簇名"接口让用户成为算法的输入源。未来算法不确定时，**给用户一个"校准通道"比强行 100% 自动更健康。**

5. **"治标 → 治本 → 观察"三段论** —— 今天解决 fastapi spawn error 是 `fuser -k`（治标）、supervisor 配置改（治本，明天做）、长期稳定性（观察）。当 bug 反复出现时，这三段要一起想，别只修一段就走。
