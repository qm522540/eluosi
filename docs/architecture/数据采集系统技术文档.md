## 二、平台客户端设计

### 2.1 抽象层设计

所有平台客户端继承 `BasePlatformClient` 抽象基类，通过 `PlatformClientFactory` 工厂实例化。

**文件**: `app/services/platform/base.py`

```
BasePlatformClient (ABC)
  ├── fetch_ad_campaigns() → list      # 拉取广告活动列表
  ├── fetch_ad_stats() → list          # 拉取广告统计数据
  ├── fetch_products() → dict          # 拉取商品列表
  ├── fetch_orders() → list            # 拉取订单数据
  ├── fetch_inventory() → list         # 拉取库存数据
  └── test_connection() → bool         # 测试API连接

PlatformClientFactory
  ├── register(platform, class)        # 注册平台实现
  └── get_client(platform, ...) → BasePlatformClient
```

**数据标准化**: 所有客户端将平台特定的数据格式转换为统一的标准格式，业务层无需关心平台差异。

### 2.2 通用安全机制

三个平台客户端共享以下安全机制：

#### 限速控制

```python
MIN_REQUEST_INTERVAL = 60.0 / rate_limit_per_minute  # 默认60次/分 → 1秒/次

async def _rate_limit(self):
    elapsed = time.monotonic() - self._last_request_time
    if elapsed < MIN_REQUEST_INTERVAL:
        await asyncio.sleep(MIN_REQUEST_INTERVAL - elapsed)
    self._last_request_time = time.monotonic()
```

每次API请求前自动计算距离上次请求的间隔，不足则等待。防止暴力调用导致被封号。

限速配置在 `app/config.py`：

| 配置项 | 默认值 | 说明 |
|--------|-------|------|
| `WB_RATE_LIMIT_PER_MINUTE` | 60 | WB每分钟最大请求数 |
| `OZON_RATE_LIMIT_PER_MINUTE` | 60 | Ozon每分钟最大请求数 |
| `YANDEX_RATE_LIMIT_PER_MINUTE` | 60 | Yandex每分钟最大请求数 |

#### 重试机制

```python
max_retries = 3

for attempt in range(max_retries):
    response = await client.request(method, url, **kwargs)
    
    if response.status_code == 429:        # 被限速
        wait_time = min(30, 5 * (attempt + 1))   # 5s → 10s → 15s
        await asyncio.sleep(wait_time)
        continue
    
    # 超时: 指数退避
    except httpx.TimeoutException:
        await asyncio.sleep(2 * (attempt + 1))    # 2s → 4s → 6s
```

| 异常类型 | 处理策略 | 退避时间 |
|---------|---------|---------|
| HTTP 429 (限速) | 等待后重试 | 5s / 10s / 15s（最大30s） |
| 超时 | 等待后重试 | 2s / 4s / 6s |
| HTTP 4xx/5xx (非429) | 立即抛出，不重试 | — |

#### 连接管理

- 使用 `httpx.AsyncClient` 长连接复用，减少TCP握手开销
- 懒加载：首次请求时创建，后续复用
- 客户端关闭后自动标记为 closed，下次请求重新创建
- 每个平台客户端提供 `close()` 方法，采集完成后调用释放资源

```python
async def _get_client(self) -> httpx.AsyncClient:
    if self._http_client is None or self._http_client.is_closed:
        self._http_client = httpx.AsyncClient(
            headers=self._get_headers(),
            timeout=httpx.Timeout(30.0, connect=10.0),
        )
    return self._http_client
```

```
POST /adv/v2/fullstats
请求体: [{"id": campaign_id, "dates": ["2026-04-06", "2026-04-07"]}]

响应结构:
[
  {
    "days": [
      {
        "date": "2026-04-06",
        "apps": [
          {
            "nm": [
              {
                "views": 1500,     → impressions
                "clicks": 45,      → clicks
                "sum": 2250.00,    → spend (RUB)
                "orders": 3,       → orders
                "ordersSumRub": 8500.00  → revenue (RUB)

          }
        ]

    ]
  }

```

**数据聚合逻辑**: 同一天内，遍历所有 `apps` → `nm`，累加 views/clicks/sum/orders/ordersSumRub。

**衍生指标计算**:

| 指标 | 公式 | 说明 |
|------|------|------|
| CTR | clicks / impressions * 100 | 点击率(%) |
| CPC | spend / clicks | 单次点击成本(RUB) |
| ACOS | spend / revenue * 100 | 广告花费占比(%) |
| ROAS | revenue / spend | 广告回报率 |

##### `fetch_ad_stats(campaign_id, date_from, date_to)` — 报表API

Yandex Direct 的统计数据通过 Reports API 获取，返回 **TSV 格式**文本，且为**异步生成**。

```
POST /reports (Direct API v5)
请求体: {
  "params": {
    "SelectionCriteria": {
      "Filter": [{"Field": "CampaignId", "Operator": "EQUALS", "Values": ["12345"]}],
      "DateFrom": "2026-04-06",
      "DateTo": "2026-04-07"
    },
    "FieldNames": ["Date", "Impressions", "Clicks", "Cost", "Conversions", "Revenue"],
    "ReportType": "CUSTOM_REPORT",
    "Format": "TSV"
  }
}
```

**异步报表生成流程**:

```
请求 → 返回状态码
  ├── 201: 报表正在生成，读取 retryIn header，等待后重试
  ├── 202: 报表还未就绪，等待10秒后重试
  └── 200: 报表就绪，响应体为TSV文本
```

**TSV 解析逻辑**:

```
第1行: 报表名称（跳过）
第2行: 列名（Date  Impressions  Clicks  Cost  ...）
第3行起: 数据行
最后一行: Total汇总行（跳过）
```

Cost 和 Revenue 字段同样为微单位，需要 ÷ 1,000,000。

### 3.3 异步桥接

Celery 任务是同步执行的，但平台客户端使用 `httpx.AsyncClient` 是异步的。通过 `_run_async()` 桥接：

```python
def _run_async(coro):
    loop = asyncio.new_event_loop()
    try:
        return loop.run_until_complete(coro)
    finally:
        loop.close()
```

每次任务执行创建新的 event loop，执行完毕后关闭。

## 四、广告数据采集任务

**文件**: `app/tasks/ad_tasks.py`

### 4.1 执行流程

以 WB 为例（Ozon/Yandex 流程相同）：

```
fetch_wb_ad_stats() 被 Celery Beat 触发
│
├── 1. 记录任务开始 → task_logs (status=running)
│
├── 2. 查询 shops 表
│   SELECT * FROM shops 
│   WHERE platform='wb' AND status='active' AND api_key IS NOT NULL
│
├── 3. 遍历每个店铺
│   │
│   ├── 3a. 调用 WBClient.fetch_ad_campaigns()
│   │   → 解析为标准格式
│   │   → _upsert_campaign(): 存在则更新，不存在则插入 → ad_campaigns
│   │
│   ├── 3b. 查询该店铺的 active/paused 活动
│   │   SELECT * FROM ad_campaigns 
│   │   WHERE shop_id=? AND status IN ('active','paused')
│   │
│   ├── 3c. 遍历每个活动，调用 WBClient.fetch_ad_stats()
│   │   → 拉取昨天+今天的数据（2天范围）
│   │   → _upsert_stat(): 按 campaign_id+stat_date+stat_hour 去重
│   │   → 写入 ad_stats
│   │
│   └── 3d. 更新 shop.last_sync_at
│
├── 4. 记录任务结束 → task_logs (status=success/failed)
│
└── 5. 返回结果 {shops_processed, campaigns_synced, stats_inserted, errors}
```

```python
    AdStat.campaign_id == campaign_id,
```

### 4.3 错误隔离

```python
for shop in shops:
    try:
        # 单店铺采集
    except Exception as e:
        errors.append(f"shop_id={shop.id}: {e}")
        db.rollback()  # 回滚该店铺的事务
        # 继续处理下一个店铺
```

单个店铺失败不阻断其他店铺的采集。所有错误汇总到 `result.errors` 列表中。

### 4.4 重试配置

```python
@celery_app.task(
    bind=True,
    max_retries=3,           # 最多重试3次
    default_retry_delay=300, # 重试间隔300秒(5分钟)
)
def fetch_wb_ad_stats(self):
    try:
        ...
    except Exception as e:
        raise self.retry(exc=e)  # 触发Celery级重试
```

**两级重试**:
1. **HTTP级**: 平台客户端的 `_request()` 方法内部重试（429/超时）
2. **任务级**: Celery 的 `self.retry()` 在整个任务失败时重试

### 4.5 三平台差异

| 客户端初始化 | `WBClient(shop_id, api_key)` | `OzonClient(shop_id, api_key, client_id=...)` | `YandexClient(shop_id, oauth_token, campaign_id=...)` |

| 统计API | `/adv/v2/fullstats` | `/api/client/statistics/daily` | Reports API (TSV异步) |

## 五、每日统计采集任务

**文件**: `app/tasks/daily_stats.py`

### 5.1 执行流程

```
fetch_wb_daily() 每天00:10触发
│
├── 1. 确定采集日期: yesterday = date.today() - 1
│
├── 2. 查询所有 active 的 WB 店铺
│
├── 3. 遍历每个店铺
│   │
│   ├── 3a. 拉取订单数据 → WBClient.fetch_orders(yesterday, yesterday)
│   │   → 过滤目标日期的订单
│   │   → 计算: 总营收、退货数、退货金额、净收入
│   │   → 去重写入 finance_revenues (按 shop_id + date)
│   │
│   └── 3b. 汇总广告花费
│       → 从 ad_stats 表查询该店铺当日广告总花费
│       → 去重写入 finance_costs (类型=ad_spend，按 shop_id + date + cost_type)
│
└── 4. 记录执行日志
```

### 5.2 数据写入

#### finance_revenues 表

| 字段 | 来源 | 计算方式 |
|------|------|---------|
| `orders_count` | 订单列表 | 总订单数 - 退货/取消数 |
| `revenue` | 订单列表 | 累加每笔订单金额(扣折扣) |
| `returns_count` | 订单列表 | isCancel=true 的订单数(WB) |
| `returns_amount` | 订单列表 | 累加退货订单金额 |
| `net_revenue` | 计算 | revenue - returns_amount |

#### finance_costs 表

| 字段 | 来源 | 说明 |
|------|------|------|
| `cost_type` | 固定 | `"ad_spend"` |
| `amount` | ad_stats表 | SUM(spend) WHERE platform=? AND stat_date=? |
| `notes` | 自动生成 | 如 "WB广告花费自动汇总(5条记录)" |

### 5.3 三平台订单数据解析差异

**WB**:
```python
total_revenue = sum(
    totalPrice * (1 - discountPercent / 100)
    for order in orders
)
returns = [o for o in orders if o.get("isCancel", False)]
```

**Ozon**:
```python
for order in orders:
    fin = order.get("financial_data", {})
    for prod in fin.get("products", []):
        total_revenue += prod.get("price", 0)
    if order["status"] in ("cancelled", "returned"):
        returns_count += 1
```

**Yandex**:
```python
for order in orders:
    items_total = sum(
        price * count
        for item in order.get("items", [])
    )
    if order["status"] in ("CANCELLED", "RETURNED"):
        returns_count += 1
```

## 六、ROI 异常检测任务

**文件**: `app/tasks/roi_alert.py`

### 6.1 检测规则

| 规则 | 条件 | 告警级别 |
|------|------|---------|
| ACOS过高 | ACOS > 30% | 警告 |
| ACOS严重过高 | ACOS > 50% | 严重 |
| ROAS过低 | ROAS < 2.0 | 警告 |
| 预算即将耗尽且回报差 | 预算用量 > 80% 且 ROAS < 1.5 | 严重 |

### 6.2 执行流程

```
check_roi_anomaly() 每30分钟触发
│
├── 遍历所有 active 店铺
│   └── 遍历该店铺所有 active 广告活动
│       │
│       ├── 查询 ad_stats 汇总今日数据
│       │   SELECT SUM(spend), SUM(revenue), SUM(orders)
│       │   FROM ad_stats WHERE campaign_id=? AND stat_date=today
│       │
│       ├── 计算 ACOS = spend / revenue * 100
│       ├── 计算 ROAS = revenue / spend
│       ├── 计算 budget_usage = spend / daily_budget
│       │
│       └── 触发告警 → 写入 notifications 表
│           type=roi_alert, channel=both
│
└── 返回 {shops_checked, alerts_generated}
```

### 6.3 告警通知格式

```
店铺: WB旗舰店 (wb)
活动: 搜索广告-001
今日花费: 15000.00 RUB
今日收入: 8500.00 RUB
订单数: 5
异常项:
  - [严重] ACOS=176.5% (阈值50%)
  - [警告] ROAS=0.57 (阈值2.0)
```

## 七、日报生成任务

**文件**: `app/tasks/report_tasks.py`

### 7.1 执行流程

```
generate_daily_report() 每天08:00触发
│
├── 确定报告日期: yesterday
│
├── 按租户分组处理
│   │
│   ├── 汇总广告数据 (ad_stats)
│   │   SELECT SUM(impressions), SUM(clicks), SUM(spend), 
│   │          SUM(orders), SUM(revenue)
│   │   FROM ad_stats WHERE tenant_id=? AND stat_date=yesterday
│   │
│   ├── 汇总营收数据 (finance_revenues)
│   │   SELECT SUM(revenue), SUM(returns_amount), SUM(net_revenue)

│   │
│   ├── 汇总费用数据 (finance_costs)

│   ├── 计算利润指标
│   │   gross_profit = total_revenue - total_cost
│   │   roi = (total_revenue - total_cost) / total_cost * 100
│   │
│   ├── 保存 ROI 快照 → finance_roi_snapshots (period=daily)
│   │
│   └── 生成通知 → notifications (type=daily_report)
│
└── 返回 {date, tenants, reports_generated}
```

## 八、数据库表操作汇总

### 读取的表

| 表 | 场景 | 查询条件 |
|----|------|---------|

### 写入的表

| 表 | 写入场景 | 去重策略 |
|----|---------|---------|
| `ad_campaigns` | 同步活动列表 | `shop_id + platform_campaign_id` |

| `finance_revenues` | 每日营收 | `shop_id + tenant_id + revenue_date` |
| `finance_costs` | 每日费用 | `shop_id + tenant_id + cost_date + cost_type` |

| `notifications` | 告警和日报通知 | 不去重，每次新增 |
| `task_logs` | 任务执行日志 | 不去重，每次新增 |
