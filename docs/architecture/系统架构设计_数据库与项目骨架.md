## 二、数据库设计（21张表）

### 全局规范
- 所有表 `tenant_id` 多租户隔离，`created_at`/`updated_at` 时间戳
- 金额 `DECIMAL(10,2)`，状态用 `ENUM`，字段 `snake_case`
- 引擎 InnoDB，字符集 `utf8mb4_unicode_ci`
- 主键 `BIGINT UNSIGNED AUTO_INCREMENT`

### 表清单

| # | 表名 | 模块 | 说明 |
|---|------|------|------|
| 1 | `tenants` | 系统 | 租户(公司/品牌) |
| 2 | `users` | 系统 | 用户，角色(owner/admin/operator/viewer) |
| 3 | `shops` | 店铺 | 三平台店铺，含API密钥(加密存储) |
| 4 | `products` | 商品 | 主商品(跨平台)，含SKU、成本价(CNY) |
| 5 | `platform_listings` | 商品 | 平台listing，关联product+shop |
| 6 | `ad_campaigns` | 广告 | 广告活动 |
| 7 | `ad_groups` | 广告 | 广告组，关联campaign+listing |
| 8 | `ad_keywords` | 广告 | 关键词(含否定词) |
| 9 | `ad_stats` | 广告 | 小时/天级广告数据(高增长表，按月分区) |
| 10 | `seo_keywords` | SEO | 关键词库 |
| 11 | `seo_templates` | SEO | 标题/描述模板 |

## 五、关键架构决策

1. **多租户隔离**: 每条SQL必须带 `WHERE tenant_id = ?`，通过 `dependencies.py` 的 `get_tenant` 从JWT提取注入
2. **平台抽象**: `services/platform/base.py` 定义抽象接口，WB/Ozon/Yandex各自实现，业务层不直接调平台API
3. **AI路由**: 所有AI调用走 `services/ai/router.py`，按任务类型自动分发模型，每次调用写 `ai_decision_logs`
4. **金额存储**: 全部RUB用 `DECIMAL(10,2)`，成本价CNY单独存，汇率换算在计算时做
5. **ad_stats增长管理**: 支持小时+天粒度，90天后小时数据聚合为天级并清理

## 六、阿里云服务器环境部署规划

### 服务器配置建议
- **ECS规格**: 2核4G起步（开发阶段），生产建议4核8G
- **系统**: Ubuntu 22.04 LTS
- **磁盘**: 系统盘40G + 数据盘100G（挂载到/data）

### 需要安装的组件
| 组件 | 版本 | 用途 |
|------|------|------|
| Python | 3.11 | 后端运行时 |
| MySQL | 8.0 | 主数据库 |
| Redis | 7.x | 缓存 + Celery消息队列 |
| Nginx | latest | 反向代理 + 静态文件 |
| Node.js | 18 LTS | 前端构建 |
| Supervisor | latest | 进程管理(FastAPI/Celery) |
| Git | latest | 代码拉取 |

### 部署架构
```
Nginx (80/443)
  ├── /api/*  → FastAPI (uvicorn :8000)
  ├── /*      → React静态文件 (/data/frontend/dist/)
  └── /ws/*   → WebSocket (预留)

Supervisor管理进程:
  ├── uvicorn (FastAPI主进程)
  ├── celery worker (任务执行)
  └── celery beat (定时调度)

MySQL (:3306) → /data/mysql/
Redis (:6379) → /data/redis/
```

### 安全配置
- 防火墙只开放 22(SSH), 80, 443
- MySQL不对外暴露，仅本地访问
- Redis设置密码，仅本地访问
- SSL证书配置(Let's Encrypt)
